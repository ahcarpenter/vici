## Summary

The project already has `prometheus-fastapi-instrumentator 7.1.0` and `prometheus-client 0.24.1` installed and wired. The `/metrics` endpoint is live. This phase is purely infrastructure plumbing: add two Docker Compose services (Prometheus + Grafana), mount config files, and add three custom metric families in the application.

The Grafana provisioning model (datasource YAML + dashboard YAML + dashboard JSON, all mounted as volumes) allows fully automated setup with zero manual steps after `docker compose up`. This is the standard pattern and well-documented.

The Inngest queue depth gauge has no external API to query (the Inngest dev server exposes no documented metrics or run-count endpoints). The best implementation is a background periodic task that polls the Inngest dev server's in-memory state—but this is not queryable. The pragmatic solution is to **omit the Inngest gauge** or implement it as a Gauge that queries the DB for a proxy metric (e.g., count of unprocessed `Message` rows). The CONTEXT.md labels this "mechanism TBD by researcher," so a DB-proxy approach or omission is the recommendation.

**Primary recommendation:** Mount `prometheus.yml`, Grafana datasource YAML, and a pre-built FastAPI dashboard JSON as volumes. Register custom metrics as module-level singletons in a new `src/metrics.py` and instrument `ExtractionService` and a background gauge collector for `pinecone_sync_queue`.

### Pattern 1: Module-level Metric Singletons

**What:** Declare `Counter`, `Histogram`, `Gauge` at module level in `src/metrics.py`. Import where needed. Avoids duplicate registration errors.
**When to use:** Always — `prometheus_client` raises `ValueError` if the same metric name is registered twice in the same process.

```python
# src/metrics.py
from prometheus_client import Counter, Histogram, Gauge

gpt_calls_total = Counter(
    "gpt_calls_total",
    "Total GPT API calls",
    ["classification_result"],
)

gpt_call_duration_seconds = Histogram(
    "gpt_call_duration_seconds",
    "GPT API call latency in seconds",
    buckets=[0.5, 1.0, 2.0, 5.0, 10.0, 30.0],
)

gpt_input_tokens_total = Counter(
    "gpt_input_tokens_total",
    "Total GPT input tokens consumed",
)

gpt_output_tokens_total = Counter(
    "gpt_output_tokens_total",
    "Total GPT output tokens consumed",
)

pinecone_sync_queue_depth = Gauge(
    "pinecone_sync_queue_depth",
    "Number of rows in pinecone_sync_queue with status=pending",
)
```

### Pattern 2: Histogram Timing via Context Manager

**What:** Use `Histogram.time()` context manager for duration recording.

```python
# src/extraction/service.py
import time
from src.metrics import gpt_calls_total, gpt_call_duration_seconds

start = time.perf_counter()
result = await self._call_with_retry(user_message)
gpt_call_duration_seconds.observe(time.perf_counter() - start)
gpt_calls_total.labels(classification_result=result.message_type).inc()
```

Note: `Histogram.time()` is a synchronous context manager. For async code, use `time.perf_counter()` manually (as above) rather than wrapping an `await` in a sync context manager.

### Pattern 3: Gauge Updater as Lifespan Background Task

**What:** Launch an `asyncio.Task` in the FastAPI lifespan that periodically queries `pinecone_sync_queue` and sets the Gauge.
**When to use:** For metrics that require DB queries (pull-based).

```python
# In lifespan(), after sessionmaker is ready:
import asyncio
from src.metrics import pinecone_sync_queue_depth

async def _update_gauges():
    while True:
        try:
            async with get_sessionmaker()() as session:
                result = await session.execute(
                    text("SELECT COUNT(*) FROM pinecone_sync_queue WHERE status = 'pending'")
                )
                pinecone_sync_queue_depth.set(result.scalar_one())
        except Exception:
            pass  # non-fatal — metric will be stale
        await asyncio.sleep(15)

_gauge_task = asyncio.create_task(_update_gauges())
yield
_gauge_task.cancel()
```

### Anti-Patterns to Avoid

- **Registering metrics inside class constructors or route handlers:** `ExtractionService.__init__` is called once per lifespan, but if tests create multiple instances, the metric is registered twice. Always register at module level.
- **Using `Histogram.time()` as context manager around `await`:** The context manager is synchronous; it does not correctly time async operations. Use `perf_counter()` manually.
- **Mounting Grafana provisioning as a single file instead of directory:** Grafana expects the provisioning path to be a directory, not individual files.

## Common Pitfalls

### Pitfall 1: Duplicate Metric Registration in Tests

**What goes wrong:** Tests that import `ExtractionService` or `src.metrics` and create multiple app instances in the same process raise `ValueError: Duplicated timeseries in CollectorRegistry`.
**Why it happens:** `prometheus_client` uses a global `REGISTRY` singleton. Re-importing or re-initializing creates duplicate registrations.
**How to avoid:** Declare all metrics at module level in `src/metrics.py`. Use `pytest` fixture that calls `REGISTRY.unregister()` if needed, or use `prometheus_client.registry.REGISTRY._names_to_collectors` to check before registering.
**Warning signs:** `ValueError: Duplicated timeseries` in pytest output.

### Pitfall 2: Prometheus Cannot Reach App Container

**What goes wrong:** Prometheus shows `app:8000` target as DOWN.
**Why it happens:** `app` service may not have started or its `/metrics` endpoint is not ready when Prometheus first scrapes.
**How to avoid:** Add `depends_on: app` to the Prometheus service. Use Prometheus's automatic retry — it will succeed on the next scrape interval. Healthcheck on app is not strictly required for this.
**Warning signs:** Target shows `connection refused` in Prometheus UI at `localhost:9090/targets`.

### Pitfall 3: Grafana Dashboard References Wrong Datasource UID

**What goes wrong:** Dashboard panels show "No data" even though Prometheus has data.
**Why it happens:** Dashboard JSON contains a hardcoded datasource UID that doesn't match the provisioned datasource's UID.
**How to avoid:** In the dashboard JSON, use `"datasource": {"type": "prometheus", "uid": "${DS_PROMETHEUS}"}` template variable, or set `"uid": "prometheus"` in the datasource provisioning YAML and reference that literal UID in the dashboard JSON.
**Warning signs:** Panel editor shows datasource as "Not found" or red.

### Pitfall 4: Token Counts Not Available from OpenAI Response

**What goes wrong:** `gpt_input_tokens_total` / `gpt_output_tokens_total` are always 0.
**Why it happens:** `client.beta.chat.completions.parse()` returns a `ParsedChatCompletion` object. Token usage is in `completion.usage.prompt_tokens` / `completion.usage.completion_tokens` — only available if not `None`.
**How to avoid:** After `_call_with_retry()` returns, also return the `completion` object (or restructure to return a tuple) so the caller can read `completion.usage`. Currently `_call_with_retry` returns only `ExtractionResult` — the method needs to also expose usage.
**Warning signs:** Token counters stay at 0 in Prometheus.

### Pitfall 5: Async Histogram Timing

**What goes wrong:** `with gpt_call_duration_seconds.time(): await ...` does not correctly measure async duration.
**Why it happens:** `Histogram.time()` uses `time.time()` and calls `__exit__` synchronously after the `await` resumes. This is actually correct for wall-clock time, but wrapping `await` inside a sync context manager works in Python (the `__exit__` fires when the `with` block exits, after `await` completes). However, the pattern is subtle and can confuse reviewers.
**How to avoid:** Use explicit `perf_counter()` for clarity and correctness.

### ExtractionService with Token Counters (Refactored Return)

```python
# src/extraction/service.py — _call_with_retry must return (result, usage)
async def _call_with_retry(self, user_message: str):
    completion = await self._client.beta.chat.completions.parse(
        model=GPT_MODEL,
        messages=[...],
        response_format=ExtractionResult,
    )
    return completion.choices[0].message.parsed, completion.usage

async def process(self, sms_text: str, phone_hash: str) -> ExtractionResult:
    import time
    from src.metrics import (
        gpt_calls_total, gpt_call_duration_seconds,
        gpt_input_tokens_total, gpt_output_tokens_total,
    )
    user_message = f"Today is {date.today().isoformat()}. Message: {sms_text}"
    with tracer.start_as_current_span("gpt.classify_and_extract") as span:
        span.set_attribute("gen_ai.system", "openai")
        span.set_attribute("gen_ai.request.model", self._settings.extraction.gpt_model)
        t0 = time.perf_counter()
        result, usage = await self._call_with_retry(user_message)
        gpt_call_duration_seconds.observe(time.perf_counter() - t0)
    gpt_calls_total.labels(classification_result=result.message_type).inc()
    if usage:
        gpt_input_tokens_total.inc(usage.prompt_tokens or 0)
        gpt_output_tokens_total.inc(usage.completion_tokens or 0)
    return result
```

## Inngest Queue Depth: Recommendation

The Inngest dev server (running as `inngest/inngest:latest` at `http://inngest:8288`) exposes no documented REST endpoint for pending run counts or queue depth. Options:

1. **Omit the Inngest gauge** — simplest, no approximation error. Inngest dev UI at `localhost:8288` already shows run history visually.
2. **DB proxy gauge** — count `Message` rows with `status='pending'` or `status='processing'` if such a status column exists. Not a direct queue depth.
3. **HTTP polling of Inngest internal API** — undocumented, fragile, not recommended.

**Recommendation:** Implement a stub `inngest_queue_depth` Gauge that is always set to 0 for now, with a comment noting no public API is available. This satisfies the CONTEXT.md requirement without introducing undocumented API coupling. The planner can make the final call.
