<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator-host → docker-engine | The operator runs `docker compose ... up -d` on a deployment host; the prod overlay file content is the contract between the human and the engine. |
| compose-file → image-registry | The image reference in the overlay points at GHCR; a malformed reference (silent `${VAR-}` expansion) would pull an unintended image or fail opaquely. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | docker-compose.prod.yml namespace | mitigate | Use literal `ghcr.io/ahcarpenter/vici` per D-05 — operators reading the file see the exact registry namespace, not an expansion that could be misconfigured. The corresponding workflow YAML uses `${{ github.repository_owner }}` lowercased per D-13 to handle repo transfers. |
| T-05-02 | Information disclosure | docker-compose.prod.yml content | accept | The file contains no secrets — only a public registry namespace and a variable substitution. Safe to commit unencrypted. |
| T-05-03 | Denial of service | GIT_SHA unset on operator host | mitigate | `${GIT_SHA:?...}` fail-loud substitution per D-03. Silent expansion to `sha-` (broken image ref) would cause `docker compose up` to attempt pulling a non-existent image and fail with an opaque registry error; fail-loud at config-parse time is a much better signal. Severity: low (no security impact, but availability/operator-experience). |
| T-05-04 | Spoofing | Image reference | mitigate | SHA-pinned tag (no `:latest`) means operators always pull a specific immutable digest; an attacker who pushed `:latest` to the registry could not affect production deploys. CI-02 invariant. |
</threat_model>
