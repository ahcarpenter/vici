    `## Image Publishing & GHCR Operator Runbook`

    Phase 5 (v1.1 De-platform milestone) publishes multi-arch Docker images to GitHub Container Registry on every push to `main`, every `v*` git tag, and on manual `workflow_dispatch`. Images are SHA-pinned (`ghcr.io/ahcarpenter/vici:sha-<7char>`); no `:latest` tag is ever applied.

    Two manual operator actions are required and CANNOT be automated by the workflow. Both must be completed by the repo owner via the GitHub web UI.

    `### Manual Step 1: Toggle GHCR Package Visibility to Public (one-time)`

    **When:** After the FIRST successful push to `main` triggers the build/merge jobs and the new GHCR package appears.

    **Why required:** GitHub creates new GHCR packages as private by default. The CI workflow's `verify` job is intentionally anonymous (no `docker/login-action`, no `packages: write` permission) so it proves the package is publicly pullable. On the first push, the `verify` job will fail with `denied: requested access to the resource is denied` — this is expected behavior and is the trigger for performing this manual step.

    **Steps:**

    1. Navigate to `https://github.com/users/ahcarpenter/packages/container/vici/settings`
    2. Scroll to the "Danger Zone" section at the bottom of the page
    3. Click "Change visibility"
    4. Select "Public"
    5. Type the package name (`vici`) to confirm and click the confirmation button
    6. Re-trigger the workflow either via CLI (`gh workflow run ci.yml --ref main`) or via the Actions tab → CI → "Run workflow" dropdown
    7. Confirm the `verify` job now passes (both platforms present, attestation manifests count is >= 2)

    **Note:** This toggle is one-way (private to public) and only needs to be done ONCE. Subsequent pushes to `main` do not re-create the package; the visibility setting persists.

    `### Manual Step 2: Update Branch Protection / Required Status Checks (pre-merge of Phase 5 PR)`

    **When:** BEFORE merging the Phase 5 pull request.

    **Why required:** The deleted `cd-base.yml`, `cd-dev.yml`, `cd-staging.yml`, and `cd-prod.yml` workflow files no longer exist after Phase 5, but the repository's branch-protection ruleset for `main` may still list `cd-prod`, `cd-staging`, `cd-dev`, or `cd-base` as required status checks. If those job names remain required, the Phase 5 PR (and every PR thereafter) will sit indefinitely in "Some checks haven't completed yet" because the workflows that produce those checks no longer exist.

    **Steps:**

    1. Navigate to Settings → Rules → Rulesets (modern UI) OR Settings → Branches → Branch protection rules (classic UI, if still in use)
    2. Open the ruleset for the `main` branch
    3. Under "Require status checks to pass before merging", REMOVE all of the following if present:
       - `cd-prod`
       - `cd-staging`
       - `cd-dev`
       - `cd-base`
    4. (Optional, recommended) ADD the new Phase 5 job names as required status checks once the first run completes successfully:
       - `test`
       - `compose-validate`
       - `build (linux/amd64)`
       - `build (linux/arm64)`
       - `merge`
       - `verify`
    5. Click Save

    **Note:** If you add `verify` as a required check immediately, the FIRST push (before Manual Step 1's public-toggle) will block the merge because verify fails on the first run. Recommended sequence: complete Manual Step 1 first, confirm verify passes, then add `verify` to required checks.

    `### Verifying Both Steps Are Complete`
