| repo → GitHub Settings | Branch-protection rules and required-status-check rulesets reference job names from workflow files; those rules live OUT of the repo and are NOT touched by `git rm`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Denial of service | branch-protection ruleset | mitigate | Plan 04 documents the manual pre-merge step (D-18) to remove `cd-prod`/`cd-staging` from required-status-check rules. Without this, Phase 5's PR sits indefinitely in "Some checks haven't completed yet" (Pitfall 6 in 05-RESEARCH.md). Severity: high (blocks merge until operator action). Mitigation lives in Plan 04. |
| T-05-06 | Information disclosure | git history | accept | The four deleted files contain GitHub Secret reference STRINGS (`secrets.WIF_PROVIDER` etc.) but not the secret values themselves — secrets are stored in GitHub Settings. Git history retains the file contents but exposes nothing sensitive. Standard risk profile for workflow-file deletion. |
| T-05-07 | Tampering | future fork PRs | mitigate | Once the files are deleted, fork PRs cannot re-introduce GKE-targeted deploys without code review. Existing CI status check enforcement on `main` (after the D-18 manual update) ensures the new `ci.yml` jobs are the merge gate. |
| T-05-08 | Elevation of privilege | GitHub Settings secrets | accept | The GitHub Settings secrets (`GCP_WIF_PROVIDER`, `GCP_CI_SA_EMAIL`, `PULUMI_CONFIG_PASSPHRASE`) remain defined in repo Settings. They become unreferenced after this deletion but are not actively misusable — no workflow consumes them. Phase 9 INFRA-01 may delete the Settings entries; not in Phase 5 scope. |
