# Phase 1.1: Apply Revised 3NF Schema and Propagate Throughout App - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate the database schema from the Phase 1 draft (phone_hash string FKs, raw_sms columns, old table names) to the fully-designed 3NF schema captured in Phase 1's CONTEXT.md. Propagate the new schema through: SQLModel models, Alembic migration, service layer, and tests. No new domain logic — only structural correctness.

Out of scope: GPT extraction, job/work_request repositories, Pinecone writes, match logic.

</domain>

<decisions>
## Implementation Decisions

### Migration strategy
- Replace (rewrite) the existing `2026-03-05_initial_schema.py` migration — do not add a second migration on top
- Nothing is deployed; clean history is the right call
- Use `sa.DateTime(timezone=True)` for all datetime columns (maps to TIMESTAMPTZ in PostgreSQL, compatible with SQLite in tests)
- Include the `match` table in this migration even though it won't be written to until Phase 3
- Delete `test.db` as part of this phase — it's a dev artifact and tests switch to in-memory SQLite

### Model file structure
- Domain folders renamed to match table names: `src/workers/` → `src/work_requests/`
- `User` model moves to a new `src/users/` domain (`src/users/models.py`)
- `src/models.py` stays as the Alembic aggregation point — update its imports to reflect new domain paths
- `Job.pay_rate` uses `Field(gt=0)` in SQLModel (application-layer validation) plus the DB CHECK constraint — defense in depth

### Service layer
- `register_phone` becomes `get_or_create_user(session, phone_hash) -> User` — returns the full User object so callers chain `user.id` for all subsequent FK writes
- `enforce_rate_limit` and `write_audit_log` accept `user_id: int` (not User object) — focused signatures
- `write_inbound_message` drops `phone_hash` and `raw_sms` parameters entirely; accepts `user_id: int` and `body: str`
- SEC-04 compliance: `audit_log.detail` stores the raw Twilio form payload as JSON (replaces `raw_sms` column on `inbound_message`)
- Function renames: `register_phone` → `get_or_create_user`

### Test changes
- `conftest.py` switches to in-memory SQLite: `sqlite+aiosqlite:///:memory:` — no file artifact, always clean
- A `make_user(session, phone_hash=...) -> User` factory fixture added to `conftest.py` — reused by all tests needing downstream rows
- `tests/sms/test_webhook.py` surgically updated: keep test structure and intent, update assertions and fixtures to new model names and FK pattern
- No new model-level constraint tests for `job`/`work_request` — that belongs to Phase 2/3

### Claude's Discretion
- Exact naming of `get_or_create_user` helper (could be `upsert_user` or `register_user`)
- Whether `make_user` fixture uses `pytest_asyncio.fixture` or a plain factory function
- Internal detail of how `UNIQUE (user_id, window_start)` is expressed in SQLModel for `rate_limit`

</decisions>

<specifics>
## Specific Ideas

- The 3NF schema is fully specified in `.planning/phases/01-infrastructure-foundation/01-CONTEXT.md` under "Database Schema (3NF — revised 2026-03-07)". The planner and researcher should treat that as the authoritative schema spec.
- `audit_log.message_id` is nullable (SET NULL on delete) — present for events after message row exists, NULL for pre-persistence events (rate limit rejection before message is written)
- `rate_limit` UNIQUE constraint was previously on `(phone_hash, window_start)` — it becomes `(user_id, window_start)` after this phase

</specifics>
