## Summary

Phase 1.1 is a structural correction phase: the Phase 1 draft schema used `phone_hash` string pseudo-FKs, `raw_sms` denormalized columns, and old table names. The fully-designed 3NF schema introduces a proper `user` identity table, integer FK relationships, and clean table naming (`message`, `work_request`, `match`). No new domain logic ships — only correctness.

The scope is well-bounded: rewrite one migration file, reorganize model files into new domain folders, update service function signatures, update the router call sites, and align tests to in-memory SQLite with updated imports. All patterns (async session, `text()` upserts, `session.add()` + `flush()`) remain the same — only the column/table names and FK topology change.

The primary risk is the cross-cutting nature: every layer touches every other layer. The planner should sequence tasks so the migration and models are locked in first (Wave 1), service layer second (Wave 2), then router and tests (Wave 3). This prevents cascading drift.

**Primary recommendation:** Rewrite the migration first, then propagate in strict dependency order: models → service → router → tests.

### Pattern 1: TIMESTAMPTZ in SQLModel

SQLModel/SQLAlchemy does not have a native `TIMESTAMPTZ` type shorthand. The correct way to express timezone-aware datetimes:

```python
# In SQLModel model
from sqlalchemy import DateTime
from sqlmodel import Field, SQLModel
from datetime import datetime, UTC
from typing import Optional
import sqlalchemy as sa

class User(SQLModel, table=True):
    __tablename__ = "user"
    id: Optional[int] = Field(default=None, primary_key=True)
    phone_hash: str = Field(unique=True)
    created_at: datetime = Field(
        default_factory=lambda: datetime.now(UTC),
        sa_column=sa.Column(sa.DateTime(timezone=True), nullable=False),
    )
```

Using `sa_column` with `sa.DateTime(timezone=True)` produces TIMESTAMPTZ in PostgreSQL and works with SQLite (SQLite ignores the timezone flag but stores the value correctly).

### Pattern 2: CHECK Constraints in SQLModel

SQLModel does not support DB-level CHECK constraints through `Field()` alone. They must be declared via `__table_args__`:

```python
from sqlalchemy import CheckConstraint
from sqlmodel import Field, SQLModel

class Job(SQLModel, table=True):
    __tablename__ = "job"
    __table_args__ = (
        CheckConstraint("pay_rate > 0", name="ck_job_pay_rate_positive"),
        CheckConstraint("estimated_duration_hours > 0", name="ck_job_duration_positive"),
    )
    pay_rate: float = Field(gt=0)          # application-layer validation
    estimated_duration_hours: Optional[float] = Field(default=None, gt=0)
```

`Field(gt=0)` applies at the Pydantic/application layer. `CheckConstraint` applies at the DB layer. Both are needed per the locked decision on defense-in-depth.

### Pattern 3: Integer FK in SQLModel

```python
from sqlalchemy import ForeignKey
import sqlalchemy as sa

class Message(SQLModel, table=True):
    __tablename__ = "message"
    user_id: int = Field(
        sa_column=sa.Column(sa.Integer, sa.ForeignKey("user.id", ondelete="RESTRICT"), nullable=False)
    )
```

Using `sa_column` with explicit `ForeignKey` is required when you need `ondelete` behavior. The simpler `Field(foreign_key="user.id")` does not support `ondelete` options.

### Pattern 4: UNIQUE constraint on multiple columns in SQLModel

```python
from sqlalchemy import UniqueConstraint

class RateLimit(SQLModel, table=True):
    __tablename__ = "rate_limit"
    __table_args__ = (
        UniqueConstraint("user_id", "created_at", name="uq_rate_limit_user_window"),
    )
```

### Pattern 6: `get_or_create_user` upsert returning User object

The existing `register_phone` uses `text()` with `ON CONFLICT DO NOTHING` and returns `None`. The new function must return the `User` ORM object so callers can chain `user.id`.

```python
async def get_or_create_user(session: AsyncSession, phone_hash: str) -> User:
    await session.execute(
        text(
            """
            INSERT INTO "user" (phone_hash, created_at)
            VALUES (:phone_hash, :created_at)
            ON CONFLICT (phone_hash) DO NOTHING
            """
        ),
        {"phone_hash": phone_hash, "created_at": datetime.now(UTC)},
    )
    result = await session.execute(
        select(User).where(User.phone_hash == phone_hash)
    )
    return result.scalar_one()
```

Note: `user` is a reserved word in PostgreSQL. The table name must be quoted as `"user"` in raw SQL. SQLModel/SQLAlchemy handles quoting automatically in ORM queries but raw `text()` statements need explicit quotes.

### Pattern 7: In-memory SQLite for tests

The current `conftest.py` uses `sqlite+aiosqlite:///./test.db` (file-based). The switch to in-memory:

```python
DATABASE_URL_TEST = "sqlite+aiosqlite:///:memory:"
```

With `scope="session"`, a single in-memory database is shared across the test session. `SQLModel.metadata.create_all` creates the schema from the current model definitions, bypassing Alembic migrations entirely for tests. This is the established pattern and remains correct after the model rename.

**Important:** `scope="session"` for `test_engine` and the in-memory URL work together. If scope were `"function"`, the in-memory DB would be destroyed between tests, requiring `create_all` for each test. The existing session-scoped approach is correct and should be preserved.

### Pattern 8: `write_audit_log` with `message_id`

After the schema change, `audit_log.message_id` is nullable (SET NULL on delete). The service function signature gains an optional `message_id` parameter:

```python
async def write_audit_log(
    session: AsyncSession,
    message_sid: str,
    event: str,
    detail: str | None = None,
    message_id: int | None = None,
) -> None:
```

For rate-limit and duplicate audit events (which fire before `message` row exists), `message_id` is `None`. For the `received` event (fired after `write_inbound_message`), the caller passes the returned message id.

### Pattern 9: Router call-site update

Current router flow:
```
phone_hash = hash_phone(from_number)
register_phone(session, phone_hash)
enforce_rate_limit(session, phone_hash)
write_inbound_message(session, message_sid, phone_hash, body, dict(form_data))
write_audit_log(session, message_sid, "received")
```

New router flow after service layer refactor:
```
phone_hash = hash_phone(from_number)
user = get_or_create_user(session, phone_hash)
is_rate_limited = enforce_rate_limit(session, user.id)
write_inbound_message(session, message_sid, user.id, body)
write_audit_log(session, message_sid, "received", detail=json.dumps(dict(form_data)), message_id=message.id)
```

The `get_or_create_user` call moves before `enforce_rate_limit` because `enforce_rate_limit` now needs `user_id`. This changes the ordering slightly from the original — the idempotency check still fires first (before any user lookup), but user registration now precedes rate limit.

**Audit log ordering issue to verify:** Currently duplicate and rate-limit audit events fire without a `user_id` or `message_id`. With the new schema, `audit_log` no longer has `user_id` — it only has `message_sid` (always present) and nullable `message_id`. So audit logging for pre-message events is still possible via `message_sid` alone. No structural problem here.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| TIMESTAMPTZ columns | Custom type | `sa.Column(sa.DateTime(timezone=True))` |
| Multi-column UNIQUE | Manual index | `UniqueConstraint` in `__table_args__` |
| DB-level CHECK | Application-only validation | `CheckConstraint` in `__table_args__` |
| ON DELETE behavior | Manual cascade logic | `sa.ForeignKey("table.id", ondelete="RESTRICT")` |
| In-memory test DB setup | File cleanup logic | `sqlite+aiosqlite:///:memory:` |

## Common Pitfalls

### Pitfall 1: `user` is a reserved word in PostgreSQL
**What goes wrong:** Raw SQL `INSERT INTO user ...` fails with syntax error in PostgreSQL.
**Why it happens:** `user` is a PostgreSQL reserved keyword (returns current user).
**How to avoid:** Always quote in raw SQL: `INSERT INTO "user" ...`. SQLAlchemy ORM and Alembic autogenerate handle quoting automatically — only `text()` statements need manual quoting.
**Warning signs:** `syntax error at or near "user"` in PostgreSQL logs; tests pass (SQLite is permissive) but production fails.

### Pitfall 2: `test.db` lingering after switch to in-memory
**What goes wrong:** Old `test.db` on disk still satisfies the old schema; some tests pass against stale data.
**How to avoid:** Delete `test.db` as part of this phase (locked decision). CI and local runs will both use `:memory:`.

### Pitfall 3: SQLModel `Field(foreign_key=...)` doesn't support `ondelete`
**What goes wrong:** `Field(foreign_key="user.id")` creates the FK but without ON DELETE RESTRICT. The migration autogenerates without the constraint, silently allowing CASCADE (PostgreSQL default is RESTRICT, but it's better to be explicit).
**How to avoid:** Use `sa_column=sa.Column(sa.Integer, sa.ForeignKey("user.id", ondelete="RESTRICT"), nullable=False)` for all FKs where delete behavior matters.

### Pitfall 4: `scope="session"` test engine with in-memory SQLite shares state
**What goes wrong:** Tests that modify data pollute later tests when the session-scoped engine is used.
**Why it happens:** In-memory DB persists for the full session; individual `async_session` fixtures rollback after each test, but only if the session wasn't committed via `session.begin()`.
**How to avoid:** The existing pattern of `await session.rollback()` in the `async_session` fixture teardown already handles this. Keep it.

### Pitfall 5: `write_audit_log` called before `message` row exists — `message_id` must be nullable
**What goes wrong:** Passing `message_id` to `write_audit_log` for duplicate/rate-limit events where no message row has been written yet.
**How to avoid:** `message_id` parameter defaults to `None` in the service function signature. Callers for pre-message events omit it. For the `received` event, `write_inbound_message` must return the `Message` object (or its id) so the router can pass it.
**Warning signs:** `IntegrityError` on audit_log insert when message_id is provided but the message row doesn't exist.

### Pitfall 6: `sa.DateTime(timezone=True)` vs naive datetime in SQLite
**What goes wrong:** SQLite does not enforce timezone-awareness. `datetime.now(UTC)` stores a timezone-aware value; SQLite reads it back as a naive string. Comparisons in tests can fail if code mixes naive and aware datetimes.
**How to avoid:** Consistently use `datetime.now(UTC)` (aware) for all `default_factory` values across all models. The existing models are inconsistent (`datetime.utcnow` in `jobs/models.py` vs `datetime.now(UTC)` in `sms/models.py`) — standardize to `lambda: datetime.now(UTC)` in all models during this phase.

### Pitfall 7: Alembic `env.py` imports stale model paths
**What goes wrong:** After renaming `src/workers/` → `src/work_requests/` and adding `src/users/`, `migrations/env.py` still imports `Worker` from `src.models` which imports from the old path. Alembic runs fine but autogenerate sees the old schema.
**How to avoid:** Update `src/models.py` imports first, then verify `migrations/env.py` transitively sees all new models via `src.models`.

### Pitfall 8: `get_or_create_user` must be called before `enforce_rate_limit`
**What goes wrong:** Current router calls `enforce_rate_limit` with `phone_hash` before registering the user. After the refactor, `enforce_rate_limit` takes `user_id: int`, which requires the user to exist first.
**Impact:** The ordering changes: `get_or_create_user` → `enforce_rate_limit`. This is correct (we must have a user to rate-limit them by user_id), but it means a new user is inserted even if they're about to be rate-limited. This is acceptable — a user row is cheap, and the rate-limit window is per-user-id anyway.
