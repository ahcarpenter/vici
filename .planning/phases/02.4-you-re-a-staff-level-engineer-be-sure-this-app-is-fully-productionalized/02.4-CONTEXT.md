<decisions>
## Implementation Decisions

### Deployment target
- **Render.com** — not Vercel. DEP-03 is redefined as Render deployment, not Vercel/Mangum.
- Render services: **Web Service** (FastAPI app) + **Managed PostgreSQL**
- Infrastructure-as-code: `render.yaml` committed to repo — defines service, env vars, pre-deploy command
- Migrations: Render **pre-deploy command** runs `alembic upgrade head` before each deploy (safe, zero-downtime pattern)
- Docker Compose `app` service **retains** `alembic upgrade head && uvicorn ...` in its `command:` override for local development continuity

### Inngest Cloud (production)
- Phase includes wiring Inngest Cloud: `INNGEST_SIGNING_KEY` and `INNGEST_EVENT_KEY` env vars referenced in `render.yaml`
- Document what needs to be created in Inngest Cloud dashboard (app registration, signing key generation)
- Production: `INNGEST_DEV=0` (not set), `INNGEST_BASE_URL` points to Inngest Cloud

### Inngest retry & error handling
- Add `retries=3` + exponential backoff to `process_message` `@create_function` decorator
- On permanent failure (retries exhausted): log at ERROR level via structlog + increment `pipeline_failures_total` Prometheus counter + visible in Inngest Cloud dashboard
- Idempotency (MessageSid dedup) is already in place — retries are safe

### sync_pinecone_queue implementation
- Implement properly: query `pinecone_sync_queue WHERE status='pending'`, attempt Pinecone upsert for each row, mark `status='success'` or `status='failed'` + increment retry count
- Currently a stub — needs real sweep logic

### Silent failure hardening
- Lifespan gauge updater: replace bare `except: pass` with `structlog.get_logger().warning(...)` so metric staleness is visible in logs

### Dockerfile hardening
- **Full multi-stage build**: builder stage (`uv sync`), runtime stage (copies app only — no build tools in production image)
- **Non-root user**: add `USER` directive in runtime stage
- **HEALTHCHECK instruction**: added to Dockerfile (uses `/health` endpoint)
- **CMD**: uvicorn only — `CMD ["uv", "run", "uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]`
- Docker Compose `app.command:` keeps the full `alembic upgrade head && uvicorn ... --reload` for local dev

### Hardening scope (non-functional only)
- **Test coverage audit**: run coverage report, identify gaps in critical paths (webhook gates, orchestrator branches), write missing tests
- **Secrets/env var documentation**: `.env.example` completeness audit, production env var checklist in README or docs, no secrets committed to docker-compose.yml
- **GitHub Actions CI**: run `pytest` + `ruff check` on push/PR; Render auto-deploys on push so CI acts as the gate

### Claude's Discretion
- Exact render.yaml service configuration details (plan size, region, health check path)
- Test coverage percentage target
- GitHub Actions workflow file structure (single job vs multi-job)
- Pinecone sync retry count / backoff strategy within sync_pinecone_queue
- Exact Prometheus counter label set for pipeline_failures_total

</decisions>
