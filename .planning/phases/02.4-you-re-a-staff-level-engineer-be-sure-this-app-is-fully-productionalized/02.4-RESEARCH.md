## Summary

This phase hardens a dev-complete FastAPI + Inngest + PostgreSQL app for production on Render.com. The six work streams are: (1) render.yaml IaC with Managed PostgreSQL and pre-deploy migrations; (2) Dockerfile multi-stage hardening; (3) Inngest retry/failure handler wiring; (4) sync_pinecone_queue real implementation; (5) test coverage audit and gap-fill; and (6) GitHub Actions CI pipeline.

The render.yaml Blueprint format is well-documented and straightforward: `preDeployCommand` for Alembic, `fromDatabase` for DATABASE_URL injection, `healthCheckPath` for the existing `/health` endpoint. Inngest Python SDK's `create_function` accepts `retries=int` and `on_failure=function` directly as decorator parameters — no custom retry logic is needed. The Dockerfile rewrite is a standard multi-stage `uv` pattern with a non-root user directive.

**Primary recommendation:** Implement the six work streams sequentially from infrastructure out — render.yaml first (enables production deployment testing), Dockerfile second (image that Render will run), then Inngest hardening, pinecone sync implementation, CI pipeline, and finally test coverage audit.

## Architecture Patterns

### render.yaml Structure

```yaml
services:
  - type: web
    name: vici
    runtime: docker
    dockerfilePath: ./Dockerfile
    preDeployCommand: "uv run alembic upgrade head"
    healthCheckPath: /health
    envVars:
      - key: DATABASE_URL
        fromDatabase:
          name: vici-db
          property: connectionString
      - key: INNGEST_SIGNING_KEY
        sync: false           # pulled from Render environment group / secret
      - key: INNGEST_EVENT_KEY
        sync: false
      - key: INNGEST_DEV
        value: "0"
      # ... all other secrets via sync: false

databases:
  - name: vici-db
    plan: basic-256mb         # Claude's discretion — start small
    postgresMajorVersion: "16"
    databaseName: vici
    user: vici
```

**Key render.yaml rules:**
- `sync: false` on secret env vars means Render prompts for them at deploy time (they are not stored in repo)
- `fromDatabase` injects the Render Managed PostgreSQL connection string automatically
- `preDeployCommand` runs AFTER the Docker build but BEFORE the container starts — correct slot for `alembic upgrade head`
- `dockerCommand` is NOT needed when CMD is set in the Dockerfile

**Render auto-deploy:** Push to the connected branch triggers a rebuild + deploy. GitHub Actions CI runs first (as gate). If CI fails, push never triggers Render deploy (Render watches the branch, not a status check) — so CI green is required to keep main deployable.

**Note on Render auto-deploy and CI gating:** Render deploys on any push to the connected branch regardless of CI status unless you configure GitHub branch protection rules. The recommended pattern: protect `main` with required status checks from GitHub Actions. Only passing CI PRs can merge to main; Render deploys from main.

### Dockerfile Multi-Stage Pattern

```dockerfile
# Builder stage
FROM python:3.12-slim AS builder
COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv
WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev

# Runtime stage
FROM python:3.12-slim AS runtime
COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv
WORKDIR /app

# Non-root user
RUN adduser --disabled-password --gecos "" appuser

# Copy only the venv and app from builder
COPY --from=builder /app/.venv /app/.venv
COPY src/ ./src/
COPY migrations/ ./migrations/
COPY alembic.ini ./

USER appuser

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

CMD ["uv", "run", "uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**Key points:**
- Builder stage runs `uv sync --frozen --no-dev` — installs deps into `.venv`
- Runtime stage copies `.venv` only — no build tools, no uv cache, smaller image
- `curl` must be available in runtime stage; add `RUN apt-get install -y --no-install-recommends curl && rm -rf /var/lib/apt/lists/*` before USER directive
- `USER appuser` must come after all COPY directives that write to `/app` (ownership)
- Docker Compose `command:` override in `app` service keeps `alembic upgrade head && uvicorn ... --reload` — this overrides Dockerfile CMD for local dev only

### Inngest Retry + Failure Handler Pattern

The Python SDK `create_function` decorator accepts `retries` and `on_failure` directly:

```python
# Source: https://www.inngest.com/docs/reference/python/functions/create
async def _handle_process_message_failure(ctx: inngest.Context) -> None:
    logger = structlog.get_logger()
    logger.error(
        "process_message: permanent failure after retries",
        message_sid=ctx.event.data.get("message_sid"),
        attempt=ctx.attempt,
    )
    from src.metrics import pipeline_failures_total
    pipeline_failures_total.inc()

@get_inngest_client().create_function(
    fn_id="process-message",
    trigger=inngest.TriggerEvent(event="message.received"),
    retries=3,
    on_failure=_handle_process_message_failure,
)
async def process_message(ctx: inngest.Context) -> str:
    ...
```

**Inngest retry behavior (HIGH confidence):**
- Default is 4 retries; setting `retries=3` means 3 retries (4 total attempts)
- Retries use exponential backoff with jitter — no custom logic needed
- `on_failure` is called once, only after all retries are exhausted
- `ctx.attempt` in on_failure context gives the attempt count
- Idempotency (MessageSid dedup already in place) makes retries safe

### sync_pinecone_queue Implementation Pattern

```python
@get_inngest_client().create_function(
    fn_id="sync-pinecone-queue",
    trigger=inngest.TriggerCron(cron="*/5 * * * *"),
)
async def sync_pinecone_queue(ctx: inngest.Context) -> str:
    logger = structlog.get_logger()
    async with get_sessionmaker()() as session:
        result = await session.execute(
            text("SELECT * FROM pinecone_sync_queue WHERE status = 'pending' LIMIT 50")
        )
        rows = result.mappings().all()

    for row in rows:
        try:
            await write_job_embedding(
                job_id=row["job_id"],
                description=row["description"],
                phone_hash=row["phone_hash"],
                openai_client=_openai_client,   # injected via module-level var like _orchestrator
                settings=get_settings(),
            )
            async with get_sessionmaker()() as session:
                await session.execute(
                    text("UPDATE pinecone_sync_queue SET status='success' WHERE id=:id"),
                    {"id": row["id"]}
                )
                await session.commit()
        except Exception as e:
            logger.warning("sync_pinecone_queue: upsert failed", job_id=row["job_id"], error=str(e))
            async with get_sessionmaker()() as session:
                await session.execute(
                    text("UPDATE pinecone_sync_queue SET status='failed', retry_count=retry_count+1 WHERE id=:id"),
                    {"id": row["id"]}
                )
                await session.commit()
    return "ok"
```

**Note:** `write_job_embedding` requires `openai_client` and `settings`. The `openai_client` must be injected as a module-level var from lifespan (same pattern as `_orchestrator`). Claude has discretion over retry count reset vs accumulate — accumulating is safer (prevents infinite retry loops).

### GitHub Actions CI Pattern

```yaml
# .github/workflows/ci.yml
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5
      - run: uv sync --frozen
      - run: uv run ruff check src/ tests/
      - run: uv run pytest tests/ -x --tb=short
```

**Claude's discretion choices made:**
- Single job (simpler, sufficient for this codebase size)
- `astral-sh/setup-uv@v5` — official uv action, fast install
- `--frozen` ensures lockfile is respected
- No coverage gate in CI (coverage audit is a separate task; gate can be added later)
- No Docker build in CI (Render builds the image; CI tests the Python code only)

### Gauge Updater Hardening Pattern

Current code in `src/main.py` lifespan:
```python
except Exception:
    pass  # non-fatal — metric will be stale until next cycle
```

Replace with:
```python
except Exception as exc:

```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Inngest retry scheduling | Custom retry loop with asyncio.sleep | `retries=N` param on create_function | Inngest handles backoff, jitter, state persistence |
| Docker layer caching for deps | Single-stage with no separation | Multi-stage builder pattern | Build tools leak into prod image; cache invalidation on code changes |
| CI environment setup | Shell scripts | `astral-sh/setup-uv` action | Handles PATH, caching, version pinning |
| DB URL injection in Render | Hardcoded connection string | `fromDatabase.property: connectionString` | Rotates automatically with DB credentials |

## Common Pitfalls

### Pitfall 1: preDeployCommand Runs in a Container Context
**What goes wrong:** `preDeployCommand` runs in the built Docker image — it must have `uv` and `alembic` available. If builder stage's `.venv` isn't copied, migrations fail silently with "command not found".
**Why it happens:** Render's pre-deploy runs `CMD`-equivalent in the image; missing runtime deps cause silent deploy failure.
**How to avoid:** Ensure `uv` binary is COPY'd into runtime stage AND `.venv` is copied from builder.
**Warning signs:** Deploy log shows "uv: not found" or "alembic: not found" during pre-deploy phase.

### Pitfall 2: Non-Root User Cannot Write to /app
**What goes wrong:** If app writes files to `/app` at runtime (logs, tmp), `USER appuser` causes permission errors.
**Why it happens:** Files COPY'd as root; appuser has no write permission.
**How to avoid:** App should write to stdout (structlog) not files. Verify no file-write paths in codebase.
**Warning signs:** `PermissionError` in startup logs.

### Pitfall 3: Inngest on_failure Handler Not Called for Step-Level Retries
**What goes wrong:** If `process_message` uses `ctx.step.run()`, step retries are independent. `on_failure` fires after function-level retries, not step retries.
**Why it happens:** Current `process_message` does NOT use step runs — it's a plain function body — so function-level `retries=3` applies to the whole function. This is fine for the current implementation.
**Warning signs:** Would only matter if steps are added later.

### Pitfall 4: render.yaml `sync: false` Requires Manual Env Var Entry on First Deploy
**What goes wrong:** First-time deploy fails with missing env vars because `sync: false` vars are not stored in render.yaml.
**Why it happens:** `sync: false` means "pull from Render dashboard, not from YAML" — correct for secrets, but operator must set them before deploying.
**How to avoid:** Document every `sync: false` var in a production env var checklist (README or docs/). Operator sets them in Render dashboard before first deploy.
**Warning signs:** App fails healthcheck immediately after first deploy; logs show `ValidationError` from pydantic_settings.

### Pitfall 5: pytest-cov Not in dev Dependencies
**What goes wrong:** `uv run pytest --cov=src` fails with "no module named pytest_cov".
**How to avoid:** Add `pytest-cov` to `[dependency-groups] dev` in pyproject.toml. Run `uv sync`.

### Pitfall 6: GitHub Actions uv Cache Miss on Every Run
**What goes wrong:** CI is slow because uv re-downloads all packages every run.
**How to avoid:** `astral-sh/setup-uv@v5` automatically handles uv cache. Alternatively add `enable-cache: true` option. Use `uv sync --frozen` (not `uv install`) to respect lockfile.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Single-stage Dockerfile | Multi-stage (builder + runtime) | Standard practice 2023+ | ~40% smaller production image; no build tools in prod |
| `CMD ["sh", "-c", "..."]` shell form | `CMD ["uv", "run", ...]` exec form | Best practice | No shell wrapper; signals propagate correctly to uvicorn |
| Run as root in container | `USER` directive (non-root) | Security standard | Limits blast radius of container escape |
| Vercel + Mangum (ASGI adapter) | Render.com Web Service (Docker) | Phase 02.5 decision | Long-lived process model; no cold starts; simpler ASGI setup |

   - What's unclear: Whether `inngest_base_url` in settings must be explicitly set to Inngest Cloud URL or can be omitted in production

2. **sync_pinecone_queue openai_client injection**
   - What we know: `write_job_embedding` requires `openai_client` and `settings`; the module-level `_orchestrator` pattern is established
   - What's unclear: Whether to add a separate `_openai_client` module-level var in `inngest_client.py` or pass the braintrust-wrapped client from lifespan
   - Recommendation: Add `_openai_client: AsyncOpenAI | None = None` module-level var, inject from lifespan alongside `_orchestrator`. Use braintrust-wrapped client for observability consistency.
