# Phase 02.1 Plan 02: Refactor ExtractionService and Normalize Repositories Summary

**One-liner:** GPT-only ExtractionService + PipelineOrchestrator owning all storage/audit/Pinecone/transaction orchestration; JobRepository and WorkRequestRepository normalized to flush-only.

## What Was Built

### ExtractionService (GPT-only)

`src/extraction/service.py` is now a pure GPT classification boundary:

- `__init__(openai_client, settings)` — receives pre-built (wrapped) client; no inline construction
- `process(sms_text, phone_hash) -> ExtractionResult` — no `session`, no `message_id`, no `user_id`
- All DB imports, `_write_audit()`, and `_enqueue_pinecone_sync()` methods removed

### PipelineOrchestrator

New `src/extraction/orchestrator.py` owns all pipeline coordination:

- Constructor receives all dependencies via DI: `extraction_service`, `job_repo`, `work_request_repo`, `message_repo`, `audit_repo`, `pinecone_client`, `twilio_client`
- `run(session, sms_text, phone_hash, message_id, user_id, message_sid, from_number)` branches by `message_type`
- **Job branch:** `job_repo.create()` (flush) → UPDATE message → audit → `session.commit()` → Pinecone (fire-and-forget after commit)
- **Worker branch:** `work_request_repo.create()` (flush) → UPDATE message → audit → `session.commit()`
- **Unknown branch:** UPDATE message → `session.commit()`
- **Pinecone failure:** caught, logged, enqueued in `pinecone_sync_queue` via separate session, pipeline does not re-raise

### Flush-only Repositories

- `JobRepository.create()`: `commit()` → `flush()`, `session.refresh()` removed
- `WorkRequestRepository.create()`: same normalization
- Caller (PipelineOrchestrator) owns the single commit

### Tests

- `test_service.py`: rewritten to inject `openai_client` directly; no session/DB tests
- `test_pipeline_orchestrator.py`: 4 tests fully implemented (no skips): job branch, worker branch, unknown branch, Pinecone failure path
- `test_persistence.py`: removed old ExtractionService-with-session tests (moved to orchestrator tests)
