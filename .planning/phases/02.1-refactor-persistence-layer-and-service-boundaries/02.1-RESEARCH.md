## Summary

Phase 2.1 is a pure refactoring phase with no new user-facing behavior. It addresses 30 staff-engineer audit findings across six categories: layering, persistence, transactions, DI, error handling, and config. The two external contracts (`process_message` Inngest function and SMS webhook router) keep identical observable behavior.

The central architectural change is splitting the monolithic `ExtractionService.process()` (which currently does GPT + DB writes + Pinecone + audit logging + commit orchestration) into two focused classes: `ExtractionService` (GPT-only) and `PipelineOrchestrator` (everything else). Repositories replace scattered inline SQL. Transaction ownership moves from each repository method to the orchestrator. Config moves from a flat `Settings` class to nested Pydantic models. Exceptions replace boolean return values.

**Primary recommendation:** Execute in six atomic waves, one category per wave. Keep all tests green on every commit by adding new abstractions before removing old code.

### Pattern 1: Repository with flush-only (target)

```python
# src/jobs/repository.py — AFTER refactor
class JobRepository:
    @staticmethod
    async def create(session: AsyncSession, job_create: JobCreate) -> Job:
        job = Job(...)
        session.add(job)
        await session.flush()   # NOT commit — caller owns transaction
        await session.refresh(job)
        return job
```

The current `JobRepository.create()` and `WorkRequestRepository.create()` both call `await session.commit()` then `await session.refresh()`. The refactor removes the commit; the orchestrator calls `await session.commit()` once per pipeline branch after all writes succeed.

### Pattern 2: ORM upsert for ON CONFLICT DO NOTHING (get_or_create_user)

```python
# src/sms/repository.py — MessageRepository.get_or_create_user
from sqlalchemy.dialects.postgresql import insert as pg_insert

stmt = pg_insert(User).values(phone_hash=phone_hash, created_at=datetime.now(UTC))
stmt = stmt.on_conflict_do_nothing(index_elements=["phone_hash"])
await session.execute(stmt)
result = await session.execute(select(User).where(User.phone_hash == phone_hash))
return result.scalar_one()
```

Note: For SQLite test compatibility, `sqlalchemy.dialects.sqlite` has its own `insert` with `on_conflict_do_nothing`. The project already uses SQLite for tests. Evaluate per-upsert whether PostgreSQL-dialect import is acceptable (tests mock the DB for service-layer tests; repository integration tests run on SQLite). Claude's discretion: use raw SQL for complex `ON CONFLICT DO UPDATE` (rate_limit) and ORM dialect insert for simple DO NOTHING (user upsert).

### Anti-Patterns to Avoid

- **Commit inside repository methods:** Repository methods call `flush()` only; orchestrator/router owns `commit()`. Exception: idempotency check in webhook router may commit separately before the Inngest emit.
- **session: Optional dual-mode APIs:** Remove the backward-compat `session=None` pattern from `ExtractionService.process()`. After decomposition, `ExtractionService` takes no session at all.
- **Inline raw SQL in service layer:** Move all `sa_text(...)` SQL into repository methods.
- **Module-level singleton construction outside lifespan:** All singleton construction (ExtractionService, PipelineOrchestrator, repositories) goes into FastAPI lifespan or `app.state`, not module-level `@lru_cache` on services.
- **Scattered `_write_audit()` copies:** `ExtractionService._write_audit()` and the standalone `write_audit_log()` in `sms/service.py` must be replaced by a single `AuditLogRepository`.

## Common Pitfalls

### Pitfall 1: Breaking SQLite test compatibility with PostgreSQL-dialect upserts
**What goes wrong:** `sqlalchemy.dialects.postgresql.insert` raises `CompileError` on SQLite. Tests use SQLite in-memory.
**Why it happens:** ORM upserts use database-specific `insert()` from dialect modules.
**How to avoid:** For the `get_or_create_user` upsert (DO NOTHING), the current raw SQL works in both SQLite and PostgreSQL. If converting to ORM, either keep raw SQL, or use a try-select-then-insert pattern that's dialect-agnostic, or accept that repository-level integration tests only run on PostgreSQL.
**Warning signs:** `CompileError: Can't render element of type <class '...OnConflictDoNothing'>` in test output.

### Pitfall 2: Double-commit after flush in refactored repositories
**What goes wrong:** Calling `session.refresh(job)` after `flush()` without a commit causes `MissingGreenlet` or stale data if the session has expired objects.
**Why it happens:** The project uses `expire_on_commit=False`, but `refresh()` after a flush (without commit) re-reads the DB for server-generated values.
**How to avoid:** After `flush()`, the ORM object already has its `id` populated (PostgreSQL returns SERIAL via RETURNING). `refresh()` is only needed after `commit()`. Remove `refresh()` from flush-only repository methods, or confirm SQLAlchemy async populates `id` from RETURNING automatically after flush.
**Warning signs:** `id` is `None` after `flush()` call.

### Pitfall 3: `session.begin()` context manager conflicts with manual flush/commit
**What goes wrong:** The SMS router currently uses `async with session.begin():` which auto-commits on exit. If inner repository methods also call commit(), a second commit is attempted.
**Why it happens:** `session.begin()` opens a transaction that commits on clean exit. Nested manual commits cause `InvalidRequestError: Can't commit — session is in a subtransaction`.
**How to avoid:** After normalizing repositories to flush-only, the `session.begin()` block in the router handles the single commit cleanly. Verify that Inngest's internal session management (currently `async with get_sessionmaker()() as session:` without `begin()`) aligns with the orchestrator's explicit commit.
**Warning signs:** `sqlalchemy.exc.InvalidRequestError: This session is in 'committed' state` in tests.

### Pitfall 4: Breaking existing Inngest tests during process_message migration
**What goes wrong:** The existing `test_process_message.py` tests mock `src.inngest_client.ExtractionService`. After migration, `process_message` calls `PipelineOrchestrator`, so the mock target changes.
**Why it happens:** Tests patch the class import path in the module where it is used.
**How to avoid:** Rewrite Inngest tests first to mock `PipelineOrchestrator` at `src.inngest_client.PipelineOrchestrator` before migrating the production code.
**Warning signs:** Tests pass locally (mock still intercepting) but new behavior is not actually tested.

### Pitfall 5: env var naming collision when nesting Settings
**What goes wrong:** Pydantic-settings nested models require env vars prefixed with the model field name when using `env_nested_delimiter`. Existing `.env` file uses flat names (`TWILIO_AUTH_TOKEN`, not `SMS__TWILIO_AUTH_TOKEN`).
**Why it happens:** Nested model env var resolution uses delimiter-separated paths.
**How to avoid:** Either rename env vars (requires updating `.env`, `docker-compose.yml`, and all test fixtures), or use Pydantic field aliases on the nested model to keep flat env var names. Claude's discretion: use aliases or a `@model_validator` to remap. Audit all env var references before changing field names.
**Warning signs:** `ValidationError: sms: field required` at startup after config refactor.

### Pitfall 6: Raw SQL audit writes in ExtractionService leave uncommitted rows
**What goes wrong:** `ExtractionService._write_audit()` currently calls `session.commit()` after each audit write. After removing commits from the service layer, an unhandled exception in the pipeline could leave audit rows that were flushed but never committed.
**Why it happens:** Moving commit ownership to the orchestrator means partial pipeline failures roll back everything including audit rows.
**How to avoid:** This is the correct behavior — audit rows should be part of the same transaction as the data they describe. The orchestrator commits atomically. If audit durability for failed pipelines is needed, a separate session or savepoint can be used, but the CONTEXT.md does not require this — keep it simple.
