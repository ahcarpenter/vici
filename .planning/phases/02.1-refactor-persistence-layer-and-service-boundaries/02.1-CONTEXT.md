# Phase 2.1: Refactor Persistence Layer and Service Boundaries - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Address all architectural debt identified in the staff-engineer audit (30 findings across layering, persistence, transactions, DI, error handling, config, and tests). The refactoring establishes consistent patterns that Phase 3 (MatchService) and beyond will build on. No new user-facing capabilities are added — behavior is preserved; structure is improved.

The two external anchor points (Inngest `process_message` function and SMS router) keep identical behavior. Their internal wiring may change.

</domain>

<decisions>
## Implementation Decisions

### Refactoring scope
- Full audit cleanup — all 30 findings across all 6 categories are in scope
- Not just the Phase 3 critical subset; full persistence, DI, config, exceptions, and test restructuring

### ExtractionService decomposition
- **Full decomposition**: split into two classes:
  - `ExtractionService` — GPT-only: classify + extract, returns `ExtractionResult`. No DB. No Pinecone.
  - `PipelineOrchestrator` — coordinates storage, Pinecone upsert, audit writes, error branching. Receives `ExtractionService`, `JobRepository`, `WorkRequestRepository`, `MessageRepository`, `AuditLogRepository` via constructor injection.
- `process_message` Inngest function calls `PipelineOrchestrator`, not `ExtractionService` directly

### Persistence layer — new repositories
- `MessageRepository` — add to `src/sms/` (or new `src/messages/` module): handles `Message` create, `message_type` update, idempotency check
- `AuditLogRepository` — centralized audit writes; replaces `_write_audit()` scattered across services
- `MessageRepository` and `AuditLogRepository` follow the same pattern as `JobRepository` and `WorkRequestRepository`

### Session / transaction management
- Session is a **required** constructor parameter or method argument everywhere — no `session: Optional` dual-mode APIs
- Transaction boundary ownership: each repository method uses `flush()` not `commit()` — the orchestrator owns the single `commit()` per pipeline branch
- Exception: `JobRepository.create()` and `WorkRequestRepository.create()` currently commit internally; normalize to flush-only, caller commits

### Raw SQL normalization
- `ON CONFLICT` upserts in `sms/service.py` (`get_or_create_user`, `enforce_rate_limit`) — convert to ORM with SQLAlchemy `insert().on_conflict_do_nothing()` / `on_conflict_do_update()` where practical
- If a specific upsert is cleaner in raw SQL (e.g., complex DO UPDATE SET), that's an acceptable exception — Claude decides case-by-case

### Dependency injection
- Constructor injection throughout — no DI framework
- `PipelineOrchestrator.__init__(extraction_service, job_repo, work_request_repo, message_repo, audit_repo, pinecone_client, twilio_client)` — all external dependencies injected
- `ExtractionService.__init__(openai_client, settings)` — receives clients, not settings-only
- App-level singletons remain on `app.state` but are constructed explicitly in FastAPI's lifespan

### Exception handling
- `RateLimitExceeded` and `DuplicateMessageSid` in `sms/exceptions.py` are raised instead of returning booleans
- Callers in SMS router updated to catch these exceptions and return appropriate HTTP responses
- Consistent with the project's existing exception pattern (`TwilioSignatureInvalid`)

### Config grouping
- `Settings` split into grouped nested Pydantic models:
  - `Settings.sms` — Twilio auth token, account SID, from number, rate limit config
  - `Settings.extraction` — OpenAI API key, GPT model name
  - `Settings.pinecone` — API key, index host
  - `Settings.observability` — Braintrust API key, OTel config
- `get_settings()` still uses `@lru_cache`; nested models are validated at startup

### Test strategy
- Tests stay **green on every commit** — refactor in atomic steps: add abstraction → migrate callers → delete old code
- Tests are **rewritten** to match the new structure: mock at new layer boundaries (mock `ExtractionService` in orchestrator tests; mock `PipelineOrchestrator` in Inngest function tests)
- **Test factories** added to `tests/conftest.py`: `make_message()`, `make_job()`, `make_work_request()` — consistent with existing `make_user()` pattern
- **3 integration tests** added covering full pipeline happy paths: `job_posting`, `worker_goal`, `unknown` — SMS webhook → Inngest event → ExtractionService → DB writes

### Anchor points
- `process_message` Inngest function and SMS router preserve **exact same behavior** (inputs, outputs, side effects)
- Internal wiring (which classes they construct and call) may change freely

### Claude's Discretion
- Exact ORM expression for each upsert (SQLAlchemy dialect nuances for ON CONFLICT)
- Order of decomposition steps within each commit
- Exact nested Settings model field names and grouping boundaries
- Tenacity retry config (unchanged from Phase 2 decisions)
- Module placement for new repositories (`src/sms/` vs. `src/messages/`)

</decisions>

## Existing Code Insights

### Reusable Assets
- `src/jobs/repository.py` — canonical repository pattern to replicate for MessageRepository and AuditLogRepository
- `tests/conftest.py:make_user()` — factory fixture pattern to replicate for make_message(), make_job(), make_work_request()
- `src/sms/exceptions.py` — `RateLimitExceeded`, `DuplicateMessageSid` already defined; just need to be raised
- `src/main.py` lifespan context manager — where new singleton construction goes

### Established Patterns
- SQLModel for table models, Pydantic BaseModel for non-table schemas
- `expire_on_commit=False` on async_sessionmaker — must be preserved
- `session.flush()` + caller-owned `commit()` is the target pattern
- Constructor injection via `__init__` — no framework
- structlog with OTel trace context — all new classes should follow this logging pattern

### Integration Points
- `src/inngest_client.py:process_message()` — calls `ExtractionService` today; will call `PipelineOrchestrator` after decomposition
- `src/main.py` lifespan — constructs `ExtractionService` singleton; will construct full DI graph
- `src/sms/router.py` — calls boolean-returning service functions today; will catch exceptions after exception normalization
