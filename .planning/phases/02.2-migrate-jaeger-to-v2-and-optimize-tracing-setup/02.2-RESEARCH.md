## Summary

Jaeger v1 (`jaegertracing/all-in-one:latest`) reached end-of-life on December 31, 2025. Jaeger v2 (current stable: 2.16.0) replaces the monolithic all-in-one container with a single configurable binary/image (`jaegertracing/jaeger`) that is deployed in separate roles via YAML configuration files. The CONTEXT.md decision to run separate collector + query containers is the correct Jaeger v2 deployment pattern.

The key architectural difference in v2 is that configuration is entirely YAML-driven (like OpenTelemetry Collector). The same `jaegertracing/jaeger` image runs as collector or query depending on the mounted config file. OpenSearch 2.x (latest stable: 2.19.4) serves as the storage backend; Jaeger v2's OpenSearch integration uses the Elasticsearch-compatible API, so configuration mirrors the Elasticsearch backend. Security must be explicitly disabled for local dev via `DISABLE_SECURITY_PLUGIN=true` (env var) or `plugins.security.disabled=true`.

For instrumentation, all four gaps (Inngest, GPT, Pinecone, Twilio) use standard OTel Python context manager spans (`with tracer.start_as_current_span(...)`). The `ALWAYS_ON` sampler is imported from `opentelemetry.sdk.trace.sampling` and passed directly to `TracerProvider(sampler=ALWAYS_ON)`. Resource attributes `deployment.environment` and `service.version` are added to the existing `Resource(attributes={...})` dict in `_configure_otel()`.

**Primary recommendation:** Use `jaegertracing/jaeger:2.16.0` for both collector and query, `opensearchproject/opensearch:2.19.4` with security disabled. Mount two separate YAML configs. Use `ALWAYS_ON` sampler in `TracerProvider`.

## Architecture Patterns

### Jaeger v2 Deployment Model

Jaeger v2 uses a single binary configurable for different roles via `--config /path/to/config.yaml`. In docker-compose:
- `jaeger-collector` runs the same `jaegertracing/jaeger` image with a collector YAML that defines OTLP receivers and OpenSearch storage exporters
- `jaeger-query` runs the same image with a query YAML that exposes port 16686 and reads from OpenSearch

### Config File Location Convention

Mount config files at `/etc/jaeger/` in containers:
- `./jaeger/collector-config.yaml` → `/etc/jaeger/collector-config.yaml`
- `./jaeger/query-config.yaml` → `/etc/jaeger/query-config.yaml`

Pass to container via command: `--config /etc/jaeger/collector-config.yaml`

### Collector YAML Template (verified from official repo)

```yaml
# Source: https://github.com/jaegertracing/jaeger/blob/main/cmd/jaeger/config-opensearch.yaml
service:
  extensions: [jaeger_storage, jaeger_query, healthcheckv2]
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [jaeger_storage_exporter]
  telemetry:
    resource:
      service.name: jaeger-collector
    logs:
      level: info

extensions:
  healthcheckv2:
    use_v2: true
    http:

  jaeger_storage:
    backends:
      main_storage:
        opensearch:
          server_urls:
            - http://opensearch:9200
          indices:
            index_prefix: "jaeger-main"
            spans:
              date_layout: "2006-01-02"
              rollover_frequency: "day"
              shards: 1
              replicas: 0

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: "0.0.0.0:4317"
      http:
        endpoint: "0.0.0.0:4318"

processors:
  batch:

exporters:
  jaeger_storage_exporter:
    trace_storage: main_storage
```

### OTel Sampler Configuration in `_configure_otel()`

```python
# Source: https://opentelemetry-python.readthedocs.io/en/latest/sdk/trace.sampling.html
from opentelemetry.sdk.trace.sampling import ALWAYS_ON

def _configure_otel(app: FastAPI) -> TracerProvider:
    settings = get_settings()
    resource = Resource(attributes={
        "service.name": settings.observability.otel_service_name,
        "deployment.environment": "development" if settings.inngest_dev else "production",
        "service.version": settings.observability.service_version,  # from GIT_SHA env var
    })
    exporter = OTLPSpanExporter(endpoint=settings.observability.otel_endpoint)
    provider = TracerProvider(resource=resource, sampler=ALWAYS_ON)
    provider.add_span_processor(BatchSpanProcessor(exporter))
    otel_trace.set_tracer_provider(provider)
    FastAPIInstrumentor().instrument_app(app)
    SQLAlchemyInstrumentor().instrument(engine=get_engine().sync_engine)
    return provider
```

### Twilio Span — wrap the full `to_thread` call

```python
# CONTEXT.md: span must wrap the full asyncio.to_thread call, not go inside the thread
with tracer.start_as_current_span("twilio.send_sms") as span:
    span.set_attribute("messaging.system", "twilio")
    span.set_attribute("messaging.destination", from_number)
    await asyncio.to_thread(
        self._twilio_client.messages.create,
        to=from_number,
        from_=settings.sms.from_number,
        body=UNKNOWN_REPLY_TEXT,
    )
```

### Anti-Patterns to Avoid

- **Putting the span inside the thread function:** The OTel context is not propagated into threads automatically. Wrapping `asyncio.to_thread()` from outside keeps the span in the async context.
- **Using env vars for sampler config instead of code:** CONTEXT.md requires sampler to be visible in code.
- **jaeger_query with replicas > 0 on single-node OpenSearch:** OpenSearch will show yellow cluster health and Jaeger may fail index creation.
- **Skipping `DISABLE_SECURITY_PLUGIN=true`:** Without it, OpenSearch 2.x starts with security enabled and requires TLS certificates; Jaeger connection fails.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OTLP receiver in Jaeger v2 | Custom OTel collector pipeline | Built-in `otlp` receiver in Jaeger v2 YAML | Jaeger v2 embeds OTel Collector — OTLP is a first-class receiver |
| Trace context propagation to Inngest | Manual header injection | `tracer.start_as_current_span()` context manager | OTel Python SDK propagates via context variables within same process |
| Sampler configuration | Custom sampler class | `ALWAYS_ON` singleton from `opentelemetry.sdk.trace.sampling` | Stable, tested, no instantiation overhead |

**Key insight:** Jaeger v2's OpenSearch integration handles all index lifecycle management (creation, rollover, aliases). Do not pre-create indices or configure index templates manually.

## Common Pitfalls

### Pitfall 1: Single-Node OpenSearch Replica Count
**What goes wrong:** Default Jaeger YAML uses `replicas: 1`. OpenSearch single-node cannot allocate replicas, leaving indices in yellow health. Jaeger may fail to write.
**Why it happens:** Official example YAMLs target multi-node clusters.
**How to avoid:** Set `replicas: 0` in all index configs for local dev.
**Warning signs:** OpenSearch `/_cluster/health` returns `"status": "yellow"`.

### Pitfall 2: OpenSearch Security Plugin Startup Time
**What goes wrong:** Jaeger collector starts before OpenSearch is ready, fails to connect, crashes and does not restart.
**Why it happens:** OpenSearch takes 15-30s to initialize, especially on first boot.
**How to avoid:** Add `healthcheck` to opensearch service + `condition: service_healthy` in jaeger-collector `depends_on`.
**Warning signs:** `jaeger-collector` exits with connection refused error immediately after starting.

### Pitfall 3: app service OTEL_EXPORTER_OTLP_ENDPOINT still pointing to old `jaeger`
**What goes wrong:** App sends traces to dead service name `jaeger`; no traces appear in Jaeger UI.
**Why it happens:** Old docker-compose had `OTEL_EXPORTER_OTLP_ENDPOINT: http://jaeger:4317`.
**How to avoid:** Update to `http://jaeger-collector:4317` when renaming the service. CONTEXT.md already calls this out.
**Warning signs:** Jaeger UI shows no traces but app logs show no OTel export errors (gRPC fails silently in BatchSpanProcessor by default).

### Pitfall 4: `gen_ai.system` attribute deprecated in favor of `gen_ai.provider.name`
**What goes wrong:** Using `gen_ai.system = "openai"` may emit a warning or be ignored in newer OTel collectors.
**Why it happens:** Semantic conventions updated; `gen_ai.provider.name` is the current attribute; `gen_ai.system` was used in pre-v1.36 conventions.
**How to avoid:** CONTEXT.md has locked `gen_ai.system = "openai"` — use it as specified. The attribute still works; Jaeger will record it. This is not a blocking issue for local dev observability.
**Warning signs:** OTel SDK may emit deprecation warnings to stderr (non-fatal).

### Pitfall 5: Twilio span inside `asyncio.to_thread` body
**What goes wrong:** Span created inside the thread cannot propagate context back to the async call chain; span appears disconnected or not recorded.
**Why it happens:** OTel Python uses `contextvars.Context` for propagation; threads don't inherit async context automatically.
**How to avoid:** As locked in CONTEXT.md, wrap the full `asyncio.to_thread(...)` call with the span, not the function inside the thread.

## Open Questions

1. **Token usage attributes on GPT spans**
   - What we know: The `braintrust.wrap_openai` wrapper intercepts the OpenAI client. The raw completion response contains `usage.prompt_tokens` and `usage.completion_tokens`.
   - What's unclear: Whether `wrap_openai` exposes usage data on the returned parsed object or if it must be read from the raw response. The `_call_with_retry` method uses `client.beta.chat.completions.parse()` and returns `completion.choices[0].message.parsed`, discarding the `usage` object.
   - Recommendation: Add `gen_ai.usage.input_tokens` / `gen_ai.usage.output_tokens` only if refactoring `_call_with_retry` to return the full completion. Otherwise, set a `span.add_event("gpt_called")` with model only and defer token attrs. This does not block phase completion.

2. **Inngest trace context propagation from webhook to function**
   - What we know: The webhook handler fires an Inngest event with `message_sid`, `from_number`, `body`. The Inngest function `process_message` runs in a separate execution context (separate HTTP call from Inngest Dev Server to `/api/inngest`).
   - What's unclear: Whether the existing trace context (from the webhook HTTP span) is propagated into the Inngest function execution. Inngest does not natively propagate OTel W3C traceparent headers through its event payload.
   - Recommendation: The Inngest span in `process_message` will start a new root trace unless trace context is manually injected into the event payload and extracted on the other side. For this phase, starting a new span at the Inngest entry point is acceptable (OBS-03 requires spans on Inngest functions, not necessarily same-trace propagation). Full end-to-end trace propagation is a separate concern.
