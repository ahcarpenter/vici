## 1. Current State Analysis

### PipelineOrchestrator.run() — Current Structure

**File:** `src/extraction/orchestrator.py`

`run()` signature:
```python
async def run(
    self,
    session: AsyncSession,
    sms_text: str,
    phone_hash: str,
    message_id: int,
    user_id: int,
    message_sid: str,
    from_number: str,
) -> ExtractionResult:
```

The method performs three sequential steps then branches:

**Step 1 — Classification (no branch):**
Calls `self._extraction_service.process(sms_text, phone_hash)` → returns `ExtractionResult`.

**Step 2 — Audit log (no branch):**
Calls `self._audit_repo.write(session, message_sid, "gpt_classified", ...)` unconditionally.

**Step 3 — Branch by `result.message_type`:**

| Branch | Condition | Actions |
|--------|-----------|---------|
| `job_posting` | `result.message_type == "job_posting" and result.job` | Build `JobCreate`, call `_job_repo.create()`, raw SQL UPDATE message type, audit write "job_created", `session.commit()`, Pinecone upsert (with retry queue on failure) |
| `worker_goal` | `result.message_type == "worker_goal" and result.worker` | Build `WorkRequestCreate`, call `_work_request_repo.create()`, raw SQL UPDATE message type, audit write "work_request_created", `session.commit()` |
| `else` (unknown) | all other cases | Raw SQL UPDATE message type to 'unknown', `session.commit()`, Twilio SMS reply via `asyncio.to_thread` |

**Return:** `result` (the `ExtractionResult`) in all three branches.

**Key observation:** Steps 1 and 2 (classify + first audit write) run before branching and are shared across all message types. Only Step 3 is branch-specific. Handlers receive everything from Step 3 onward.

### Repository Pattern — Current Implementation

All 4 repositories use `@staticmethod`. The pattern is structurally identical: construct ORM object, `session.add()`, `await session.flush()`, `return` (or nothing for `AuditLogRepository`).

**JobRepository.create()** — returns `Job`:
```python
@staticmethod
async def create(session: AsyncSession, job_create: JobCreate) -> Job:
    job = Job(...)   # 12-field construction with datetime parsing logic
    session.add(job)
    await session.flush()
    return job
```
Has non-trivial pre-construction logic (ISO datetime parsing, timezone normalization).

**WorkRequestRepository.create()** — returns `WorkRequest`:
```python
@staticmethod
async def create(session: AsyncSession, wr_create: WorkRequestCreate) -> WorkRequest:
    wr = WorkRequest(...)  # 5-field construction
    session.add(wr)
    await session.flush()
    return wr
```
Minimal construction logic — straightforward field mapping.

**MessageRepository.create()** — returns `Message`:
```python
@staticmethod
async def create(session: AsyncSession, message_sid: str, user_id: int, body: str) -> Message:
    message = Message(...)  # 3-field construction
    session.add(message)
    await session.flush()
    return message
```
Takes individual keyword args (not a schema object) unlike the other two. Has additional non-persist static methods (`check_idempotency`, `enforce_rate_limit`) that must NOT be affected.

**AuditLogRepository.write()** — returns `None`:
```python
@staticmethod
async def write(session: AsyncSession, message_sid: str, event: str,
                detail: str | None = None, message_id: int | None = None) -> None:
    row = AuditLog(...)
    session.add(row)
    await session.flush()
    # no return
```
Only difference from the others: returns `None`, not the entity.

**UserRepository** — excluded. Uses raw SQL only (`text()` + `SELECT`). No `session.add()` pattern.

## 2. Technical Approach

### Chain of Responsibility Implementation

**Recommendation:** Implement `MessageHandler` ABC in `src/pipeline/handlers/base.py`. Each concrete handler lives in its own file.

**Handler context parameter decision (Claude's Discretion):**
Use a `dataclass` for the context object. The `handle()` method needs: `session`, `result`, `sms_text`, `phone_hash`, `message_id`, `user_id`, `message_sid`, `from_number`. Passing a dataclass is cleaner than 8 positional args and makes Phase 3/4 handler additions non-breaking (new fields added to the dataclass without touching existing handler signatures).

Recommended context dataclass (lives in `src/pipeline/context.py`):
```python
@dataclass
class PipelineContext:
    session: AsyncSession
    result: ExtractionResult
    sms_text: str
    phone_hash: str
    message_id: int
    user_id: int
    message_sid: str
    from_number: str
```

**Handler ABC:**
```python
from abc import ABC, abstractmethod
from src.pipeline.context import PipelineContext
from src.extraction.schemas import ExtractionResult

class MessageHandler(ABC):
    @abstractmethod
    def can_handle(self, result: ExtractionResult) -> bool: ...

    @abstractmethod
    async def handle(self, ctx: PipelineContext) -> None: ...
```

**Handler chain wiring (Claude's Discretion):**
Recommend constructor injection into `PipelineOrchestrator`. This keeps `main.py` as the single place where the DI graph is assembled, which is already the established pattern. `PipelineOrchestrator.__init__` receives `handlers: list[MessageHandler]` instead of the individual repo/client parameters.

Dependencies flow to handlers at construction time in `main.py`:
- `JobPostingHandler(job_repo=JobRepository, audit_repo=AuditLogRepository, pinecone_client=write_job_embedding, extraction_service=extraction_service)`
- `WorkerGoalHandler(work_request_repo=WorkRequestRepository, audit_repo=AuditLogRepository)`
- `UnknownMessageHandler(audit_repo=AuditLogRepository, twilio_client=twilio_client, extraction_service=extraction_service)`

**PipelineOrchestrator.run() after refactor:**
```python
async def run(self, session, sms_text, phone_hash, message_id, user_id, message_sid, from_number) -> ExtractionResult:
    result = await self._extraction_service.process(sms_text, phone_hash)
    await self._audit_repo.write(session, message_sid, "gpt_classified", ...)

    ctx = PipelineContext(session=session, result=result, sms_text=sms_text,
                          phone_hash=phone_hash, message_id=message_id,
                          user_id=user_id, message_sid=message_sid, from_number=from_number)

    for handler in self._handlers:
        if handler.can_handle(result):
            await handler.handle(ctx)
            break

    return result
```

Note: `self._extraction_service` and `self._audit_repo` are still needed on the orchestrator directly for steps 1 and 2 (pre-branch work). They should NOT be removed from the orchestrator, but also must be passed to handlers that need them.

Alternatively, steps 1 and 2 could be moved into a pre-handler or the first handler — but the cleaner approach is to keep shared pre-branch logic on the orchestrator and pass extraction_service/audit_repo into handlers that additionally need them.

**File layout:**
```
src/pipeline/
├── __init__.py
├── context.py          # PipelineContext dataclass
└── handlers/
    ├── __init__.py
    ├── base.py         # MessageHandler ABC
    ├── job_posting.py  # JobPostingHandler
    ├── worker_goal.py  # WorkerGoalHandler
    └── unknown.py      # UnknownMessageHandler
```

This matches D-02. One handler per file is preferred over a single `handlers.py` because Phase 3 will add `match.py` without touching existing files.

### BaseRepository Template Method

**Recommendation:** `BaseRepository._persist` as an instance method (not staticmethod, not classmethod).

Rationale: The 4 repositories currently use `@staticmethod`. To call `self._persist(session, entity)` from within the create/write methods, those methods must convert from `@staticmethod` to instance methods. This is required regardless — D-11 acknowledges this and leaves the call to the planner. Instance method is the simplest approach: `self._persist(session, entity)` reads clearly and requires no metaclass gymnastics.

`AuditLogRepository.write()` returns `None` rather than the entity. `_persist()` should return the entity (`T`) so callers that need it get it, and callers that don't (AuditLog) simply discard the return value.

**BaseRepository in `src/repository.py`:**
```python
from abc import ABC
from typing import TypeVar
from sqlalchemy.ext.asyncio import AsyncSession

T = TypeVar("T")

class BaseRepository(ABC):
    async def _persist(self, session: AsyncSession, entity: T) -> T:
        session.add(entity)
        await session.flush()
        return entity
```

**Subclass pattern — all 4 repos:**
- Drop `@staticmethod` decorator from the `create`/`write` method
- Replace the 3-line `session.add / flush / return` block with `return await self._persist(session, entity)`
- `AuditLogRepository.write()` calls `await self._persist(session, row)` and does not return it

**Impact on main.py DI:**
Currently all repos are passed as class references and called as `self._job_repo.create(session, ...)`. Once methods become instance methods, `JobRepository()` (an instance) must be passed instead of `JobRepository` (the class). This is a required change in `main.py` for all 4 repos that gain instance methods. `UserRepository` is unaffected.

## 3. Key Risks / Gotchas

### Async throughout — no sync/async mixing concern
All handler `handle()` methods will be `async def` because every branch in the current code contains at least one `await` (session operations, Twilio `asyncio.to_thread`, Pinecone client). The `MessageHandler.handle()` ABC must declare `async def handle()`. `can_handle()` is synchronous (pure discriminator logic on an already-resolved `ExtractionResult`).

### Repositories change from class-ref to instance injection
All 4 repositories being refactored will change from `@staticmethod` to instance methods. Every call site that currently passes the class (e.g., `job_repo=JobRepository`) must be updated to pass an instance (`job_repo=JobRepository()`). The only current wiring location is `main.py` `lifespan`. If handlers receive repos at construction time, the instance wiring belongs in the handler constructors inside `lifespan`. One place to change.

### message_repo is injected but unused in run()
`MessageRepository` is currently passed to `PipelineOrchestrator` but `run()` never calls it. After refactor, `message_repo` should be dropped from `PipelineOrchestrator.__init__` entirely since no handler needs it either (the message row is already created before `run()` is called, in the webhook handler). Confirm by searching for any other usage of `self._message_repo` in the orchestrator.

### UnknownMessageHandler needs extraction_service for settings
The unknown branch accesses `self._extraction_service.settings.sms.from_number`. `UnknownMessageHandler` must receive `extraction_service` (or just `settings`) in its constructor. The cleanest approach: pass `extraction_service` since it's already a dependency in the existing orchestrator.

### Pinecone error handling — retry queue requires get_sessionmaker()
The Pinecone failure fallback in `JobPostingHandler` opens a second session via `get_sessionmaker()`. This is acceptable and should remain as-is. `get_sessionmaker()` is a module-level import from `src.database` — no additional injection needed.

### inngest_client.py call site is unchanged
The `orchestrator.run(session=..., sms_text=..., ...)` call in `inngest_client.py` uses keyword arguments matching the exact current signature. This signature MUST NOT change. The refactor is internal to `run()` only.

### AuditLogRepository.write() — return type differs
Three repos return the entity; `AuditLogRepository.write()` returns `None`. `BaseRepository._persist()` returns `T`. The write() method simply does not propagate the return: `await self._persist(session, row)` without `return`. This is fine and keeps the existing `-> None` annotation.

## 4. Implementation Order

Suggested ordering to minimize risk and keep each task independently testable:

1. **Create `src/repository.py`** — `BaseRepository` with `_persist()`. No existing code changes. Standalone new file. Verifiable in isolation.

2. **Migrate 4 repositories to subclass `BaseRepository`** — Convert `@staticmethod` methods to instance methods, replace 3-line persist block with `_persist()` call. Update `main.py` to pass instances instead of class references. Verify existing behavior unchanged (no logic change, only structural).

3. **Create pipeline domain skeleton** — `src/pipeline/__init__.py`, `src/pipeline/context.py` (PipelineContext dataclass), `src/pipeline/handlers/__init__.py`, `src/pipeline/handlers/base.py` (MessageHandler ABC). No existing code changes. Pure new code.

4. **Implement three concrete handlers** — `JobPostingHandler`, `WorkerGoalHandler`, `UnknownMessageHandler`. Extract logic verbatim from `PipelineOrchestrator.run()` branches. Handler constructors receive the dependencies they need.

5. **Refactor `PipelineOrchestrator`** — Replace branch logic with handler loop. Remove individual repo/client params from `__init__`, add `handlers: list[MessageHandler]` param. Keep `extraction_service` and `audit_repo` on orchestrator for pre-branch steps 1 and 2.

6. **Update `main.py` DI wiring** — Construct handler instances with their dependencies, pass list to `PipelineOrchestrator`. Pass repo instances (not classes) to handlers.

Steps 1–2 (BaseRepository) and steps 3–4 (pipeline skeleton + handlers) can be executed in parallel if desired since they touch different parts of the codebase. Step 5 depends on step 4. Step 6 depends on steps 2 and 5.

## Project Constraints (from CLAUDE.md / AGENTS.md)

- **SOLID application rule:** Apply only when near-term churn is expected OR 3+ instances of the same code. Both conditions are satisfied here (Phase 3/4 will add branches; 4 repos share identical persist pattern). Do NOT over-engineer beyond what is required for these two targets.
- **GoF patterns as structural vehicle:** Chain of Responsibility for orchestrator; Template Method for repositories.
- **async def routes / methods:** All handler `handle()` methods must be `async def`. `can_handle()` is sync.
- **Sync library in async context:** Twilio client uses `asyncio.to_thread()` — this must be preserved in `UnknownMessageHandler`.
- **ruff:** All new and modified files must pass `ruff check --fix src` and `ruff format src`.
- **Domain-organized structure:** Handler classes belong in `src/pipeline/handlers/`, not in `src/extraction/`.
- **Import convention:** Use explicit module names when importing across domains (e.g., `from src.pipeline.handlers.base import MessageHandler`).
