<decisions>
## Implementation Decisions

### PipelineOrchestrator — Chain of Responsibility

- **D-01:** Refactor `PipelineOrchestrator.run()` from 3 inline branches into a Chain of Responsibility pattern. Phase 3 adds match logic and Phase 4 adds outbound SMS — both will require new branches. OCP violation if left inline.
- **D-02:** Handler classes live in `src/pipeline/handlers/`. This makes the pipeline a first-class domain concept.
- **D-03:** Handler interface: Abstract base class with two methods:
  - `can_handle(result: ExtractionResult) -> bool` — decides if this handler owns the message type
  - `handle(session, result, context) -> None` — executes the domain logic for that type
- **D-04:** `PipelineOrchestrator` iterates the handler chain; the first handler where `can_handle()` returns True is invoked. No handler claiming it → fallback to unknown handler (which must always be last in the chain).
- **D-05:** Three initial concrete handlers: `JobPostingHandler`, `WorkerGoalHandler`, `UnknownMessageHandler`.
- **D-06:** GoF pattern: Chain of Responsibility. Each handler is independent and doesn't know about others.

### Repository base class — Template Method

- **D-07:** Extract the repeated `session.add(entity) + await session.flush() + return entity` pattern into a `BaseRepository` abstract base class with a `_persist(session, entity)` protected method.
- **D-08:** 4 of 5 repositories share this pattern: `JobRepository`, `WorkRequestRepository`, `MessageRepository` (`.create()`), `AuditLogRepository` (`.write()`). `UserRepository` uses raw SQL only — does not need to subclass.
- **D-09:** `BaseRepository` lives in `src/repository.py` (new top-level file, keeps `database.py` focused on engine/session config).
- **D-10:** GoF pattern: Template Method. `BaseRepository._persist()` defines the flush contract; subclasses implement domain-specific creation logic, then call `_persist()`.
- **D-11:** Static methods in subclasses become instance methods IF needed to access `self`, otherwise they may remain static if the only change is calling a module-level or classmethod `_persist`. Planner to decide the cleanest approach.

### GoF alignment principle
- **D-12:** GoF design patterns are the structural mechanism for applying SOLID. When SOLID requires abstraction, reach for the appropriate GoF pattern (COR, Strategy, Template Method, etc.) rather than ad-hoc abstraction.

### Claude's Discretion
- Exact method signature for `handle()` context parameter (dict vs dataclass vs explicit kwargs)
- Whether `BaseRepository._persist` is a classmethod, staticmethod, or instance method
- File naming within `src/pipeline/handlers/` (e.g., `job_posting.py` vs `handlers.py` with all classes)
- Whether `PipelineOrchestrator` receives the handler chain via constructor injection or builds it internally

### Reviewed Todos (not folded)
- **Refactor code such that Temporal is able to be leveraged** — out of scope; Inngest abstraction is a larger architectural change for a dedicated phase
- **Ensure tusk is leveraged** — not SOLID-related; deferred for evaluation in a tooling phase

</decisions>

## Existing Code Insights

### Reusable Assets
- `src/extraction/schemas.py` — `ExtractionResult` with `message_type` discriminator; handlers use this for `can_handle()` checks
- `src/extraction/orchestrator.py` — All handler dependencies already injected; constructor can pass them to handler instances
- `src/sms/exceptions.py` — Exception hierarchy already established; handlers may raise these

### Established Patterns
- All repositories use `@staticmethod` — if BaseRepository._persist is a classmethod or instance method, repositories will need to shift to instance methods or use a classmethod approach
- `session.add(entity) + await session.flush() + return entity` — the 4-line pattern to extract
- Chain of Responsibility canonical: `can_handle()` → `handle()` per handler, orchestrator loops

### Integration Points
- `src/inngest_client.py` — Calls `orchestrator.run()` — interface must remain unchanged
- `src/main.py` — Builds DI graph including `PipelineOrchestrator`; handler chain wiring belongs here or in a factory on `PipelineOrchestrator`
- Phase 3 `MatchService` will add a 4th handler (`MatchHandler`) — the chain must be extensible without modifying existing handlers

</code_context>
