## Current State (discovered during planning)

DNS is propagated:
```
dig dev.usevici.com +short  -->  34.120.195.235
```

App pod is running:
```
kubectl get pods -n vici  -->  vici-app-675f9554d6-74c9z  2/2 Running
```

Certificate is NOT ready:
```
kubectl get certificate -n vici  -->  vici-tls  READY=False
```

Root cause: There are 3 MutatingWebhookConfigurations for cert-manager from different Helm releases:
- `cert-manager-4d0776c0-webhook` (STALE - points to non-existent service)
- `cert-manager-d160d62d-webhook` (STALE - points to non-existent service)
- `cert-manager-554d5588-webhook` (CURRENT - the active Helm release)

The stale `4d0776c0` webhook intercepts CertificateRequest creation and tries to call a service that does not exist, causing the error:
```
Failed to create CertificateRequest: Internal error occurred: failed calling webhook
"webhook.cert-manager.io": Post "https://cert-manager-4d0776c0-webhook.cert-manager.svc:443/mutate?timeout=30s":
service "cert-manager-4d0776c0-webhook" not found
```

Additionally, the GKE load balancer reports:
```
Error 404: sslCertificates/k8s2-cr-... was not found
```
This is because the TLS secret placeholder has empty tls.crt/tls.key data and GKE cannot create an SSL certificate from it.
</context>

<task type="auto">
  <name>Task 1: Clean up stale cert-manager webhooks and re-trigger certificate issuance</name>
  <files></files>
  <action>
1. Delete the stale MutatingWebhookConfigurations that point to non-existent cert-manager services:
   ```
   kubectl delete mutatingwebhookconfiguration cert-manager-4d0776c0-webhook
   kubectl delete mutatingwebhookconfiguration cert-manager-d160d62d-webhook
   ```

2. Verify only the current webhook remains:
   ```
   kubectl get mutatingwebhookconfigurations | grep cert
   ```
   Should show only `cert-manager-554d5588-webhook`.

3. Delete the existing broken Certificate resource so cert-manager re-creates it from the ingress annotation:
   ```
   kubectl delete certificate vici-tls -n vici
   ```

4. Delete the placeholder TLS secret so cert-manager can create a fresh one with real data:
   ```
   kubectl delete secret vici-tls -n vici
   ```

5. Wait up to 60 seconds, then check that cert-manager has re-created the Certificate:
   ```
   kubectl get certificate -n vici
   ```
   The Certificate should appear (may still show READY=False while ACME challenge is in progress).

6. Check for CertificateRequest and Challenge resources (confirms the webhook is no longer blocking):
   ```
   kubectl get certificaterequests -n vici
   kubectl get challenges -n vici
   ```

7. If the Certificate does not re-appear after deleting, trigger re-creation by annotating the ingress (forces cert-manager ingress-shim to re-process):
   ```
   kubectl annotate ingress vici-ingress -n vici cert-manager.io/issuer=letsencrypt-staging --overwrite
   ```

8. Run `pulumi up --yes --skip-preview` from the `infra/` directory to reconcile any Pulumi state drift from the manual kubectl deletions. This will re-create the TLS secret placeholder if needed.

NOTE: Do NOT delete the current webhook `cert-manager-554d5588-webhook` -- that is the active one.
