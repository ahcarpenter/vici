## Gaps

The first CD Dev pipeline run against a real dev stack exposed five downstream gaps not caught by the Phase 05.1 static tests. All five were fixed as hot-fix commits on `main` in this same session. They are recorded here for traceability and as lessons for future phases.

In the same session, the three warning-level findings from the initial Phase 05.1 code review (05.1-REVIEW.md) were also closed as hot-fix commits, with new static tests added to lock the invariants and prevent regression. Those fixes are recorded below the gaps as `review-closure`.

### gap-1: PULUMI_CONFIG_PASSPHRASE secret held stale value
- surfaced: run 24290661157, step `pulumi/actions@v6`
- error: `get stack secrets manager: incorrect passphrase`
- cause: the secret set during Phase 5 (2026-04-06) did not match the passphrase currently encrypting the dev stack config.
- fix: re-set `PULUMI_CONFIG_PASSPHRASE` via `gh secret set PULUMI_CONFIG_PASSPHRASE` (2026-04-11)
- status: resolved

### gap-2: infra/ had no pyproject.toml or uv.lock
- surfaced: run 24290661157 (rerun), step `pulumi/actions@v6`
- error: `ModuleNotFoundError: No module named 'pulumi'`
- cause: `cd infra && uv sync --frozen` fell through to the root `pyproject.toml` (FastAPI app deps only) because no infra-scoped project file existed. The local `infra/.venv` was an ad-hoc manual setup that had never been formalized.
- fix: created `infra/pyproject.toml` (commit d30d3f9) pinning `pulumi`, `pulumi-gcp`, `pulumi-kubernetes` with `package = false`; generated `infra/uv.lock`
- status: resolved

### gap-3: loose pulumi-gcp version pin triggered provider drift
- surfaced: run 24290783775, step `pulumi/actions@v6`
- error: `compute.instanceGroupManagers.list forbidden` during a cluster refresh
- cause: initial `infra/pyproject.toml` used `pulumi-gcp>=9.18.0,<10`, so `uv lock` picked 9.19 which added a new `autopilotPrivilegedAdmissions` cluster field. The resulting implicit update needed compute API reads that `roles/container.developer` does not grant.
- fix: exact-pin to `pulumi-gcp==9.18.0` to match the schema version that wrote the current stack state (commit 4b80265)
- status: resolved

### gap-4: CI service account missing compute.viewer role
- surfaced: run 24290783775, step `pulumi/actions@v6`
- error: `compute.instanceGroupManagers.list forbidden`
- cause: `vici-ci-push` had `storage.objectAdmin + container.developer + artifactregistry.writer` but not `compute.viewer`. `container.developer` covers the Kubernetes control plane but not the underlying Compute Engine primitives. Any future provider upgrade or cluster refresh would hit 403.
- fix: added `ci_compute_viewer` IAMMember binding in `infra/components/cd.py` granting `roles/compute.viewer` (commit 4b80265); applied to dev stack via local `pulumi up`
- status: resolved

### gap-5: gke-gcloud-auth-plugin not installed on runner
- surfaced: run 24290885283, step `pulumi/actions@v6`
- error: `configured Kubernetes cluster is unreachable: exec: executable gke-gcloud-auth-plugin not found`
- cause: the Pulumi Kubernetes provider in `infra/` reads a kubeconfig whose auth stanza delegates to the `gke-gcloud-auth-plugin` exec plugin (standard GKE auth). GitHub's ubuntu-latest runner ships `gcloud` but not this plugin component.
- fix: added `google-github-actions/setup-gcloud@v2` step with `install_components: gke-gcloud-auth-plugin` in the `cd-base.yml` deploy job (commit b0714e1)
- status: resolved

## Review Closures

### review-closure-WR-01: prod approval gate did not cover image push
- surfaced: 05.1-REVIEW.md warning WR-01
- issue: `environment: ${{ inputs.environment }}` was only on the deploy job; anyone with workflow_dispatch permission on cd-prod.yml could push arbitrary bytes to the prod Artifact Registry before the approval gate fired.
- fix: attach `environment: ${{ inputs.environment }}` to the cd-base.yml build job so the gate covers both jobs (commit e96c95d). Empty string for dev/staging is a no-op; `prod` binds to the GitHub `prod` Environment.
- lock: new static test `TestCDBaseStructure::test_cd_base_build_job_has_environment_gate` verified.
- runtime-validated: CD Dev run 24291291708 confirmed the empty-string case passes GitHub Actions validation (build job completed successfully for `environment: ''`).
- status: resolved

### review-closure-WR-02: no concurrency control on deploys
- surfaced: 05.1-REVIEW.md warning WR-02
- issue: two quick pushes / dispatches could race two `pulumi up` calls against the same GCS-backed stack lock, producing lock contention or interleaved updates.
- fix: add top-level `concurrency:` groups to cd-dev.yml (group: cd-dev, cancel-in-progress: true), cd-staging.yml and cd-prod.yml (cancel-in-progress: false so approved deploys are never silently cancelled mid-apply) (commit e96c95d).
- lock: new parametrized static test `TestCallerConcurrency::test_caller_has_concurrency_group` verifies all three callers have the expected group.
- status: resolved

### review-closure-WR-03: short health check budget, no triage on failure
- surfaced: 05.1-REVIEW.md warning WR-03
- issue: 60s health check budget was too short to cover image pull + rolling update + readiness initial delay; on failure there was no pod-level diagnostic data in the log, requiring manual cluster access for triage.
- fix: extend health check to 36 iterations × 5s = 180s. On failure, run `gcloud container clusters get-credentials` and dump `kubectl describe deployment`, `kubectl get pods`, and `kubectl logs --tail=100 --all-containers` inline (commit e96c95d). gke-gcloud-auth-plugin is already installed on the runner from gap-5 fix.
- lock: new static test `TestCDBaseStructure::test_cd_base_health_check_extended_budget` verifies the loop has ≥36 iterations and references `kubectl`.
- runtime-validated: CD Dev run 24291291708 health check passed on attempt 2 (~10s into the 180s budget, well within the old budget too — the extended budget is insurance for slower rollouts).
- status: resolved

### review-closure-IN-01: caller workflow permissions (reopened as review-error)
- surfaced: 05.1-REVIEW.md info IN-01 — claimed the top-level `permissions:` blocks on cd-dev/staging/prod were dead config because reusable workflow jobs supposedly inherit from the called workflow.
- attempted fix: removed the `permissions:` block from all three callers (commit 74cd75f).
- outcome: **CD Dev run 24291482351 failed at workflow startup** (`startup_failure`, no jobs, ~1s, no logs, no annotations). Local actionlint and PyYAML both accepted the file. The failure was GitHub-server-side workflow validation.
- diagnosis: the original review finding is incorrect. When a caller uses `uses: ./.github/workflows/cd-base.yml`, the caller's workflow-level `permissions:` forms the UPPER BOUND for what the called workflow's jobs can claim. Without an explicit grant the caller defaults to `contents: read` only, and the called workflow's `id-token: write` request is capped out, which GitHub enforces at startup time.
- final fix: restored the `permissions: { contents: read, id-token: write }` block on all three caller workflows with an inline comment documenting why it's required (commit 717eee2). Runtime-validated by CD Dev run 24291614628 (success, ~2m45s).
- status: resolved — closed as "review-error". The caller-level permissions block IS necessary.

### review-closure-IN-02: config-map JSON injection fragility (deferred)
- surfaced: 05.1-REVIEW.md info IN-02 — single-quoted JSON config-map is fragile to quote/brace injection if the image tag source ever broadens beyond `[0-9a-f]{7}`.
- attempted fix: rewrote the config-map as a YAML block scalar `|` (commit 74cd75f).
- outcome: CD Dev run 24291482351 failed at startup_failure. Config-map reversion to JSON in commit 7835bcd still failed startup, which ruled config-map out as the primary cause — the true cause was the simultaneous IN-01 permissions removal (see review-closure-IN-01). After restoring permissions in 717eee2, the JSON form works fine (confirmed in run 24291614628).
- decision: leave the config-map in JSON form for now. Re-attempting the YAML block form would need a GitHub Actions-aware test harness (actionlint is insufficient) so we can catch startup failures before pushing. The underlying fragility is theoretical — `needs.build.outputs.sha` can't produce unsafe characters today.
- status: deferred — tracked here as a future-work item, not closed.
