- D-11: Required GitHub repo secrets: `GCP_WIF_PROVIDER`, `GCP_CI_SA_EMAIL` (CI service account email for WIF).

## Summary

Phase 5.1 delivers four GitHub Actions workflow files plus two Pulumi Python changes. The workflow files and the Pulumi infra resources (WIF pool, provider, CI service account) already exist, but the current implementation diverges from the locked Phase 5.1 decisions in five areas that require correction.

The most important structural change is splitting `cd-base.yml` from a single-job workflow into two jobs (`build` and `deploy`), enabling the deploy job to be retried independently of the build. A secondary code change adds `IMAGE_TAG` to `infra/config.py` and updates `infra/components/app.py` to use it — this enables `--config app:imageTag=<sha>` CLI flags from CI to drive the deployed image version.

The existing `cd-base.yml` already has the correct WIF authentication pattern using `google-github-actions/auth@v3` and `pulumi/actions@v6` with `PULUMI_BACKEND_URL`. The `PULUMI_CONFIG_PASSPHRASE` secret is required to decrypt stack secrets (each stack has `encryptionsalt`); this is correct in the existing code but not explicitly called out in CONTEXT.md decisions — it must be retained.

**Primary recommendation:** Rewrite all four CD workflow files from scratch against the locked decisions, and add `IMAGE_TAG = cfg.get("imageTag") or ENV` to `config.py` + update `app.py` line 122 to use `IMAGE_TAG`.

## Gap Analysis: Current Code vs. CONTEXT.md Decisions

This section documents every divergence between the existing `.github/workflows/cd-*.yml` files and the locked Phase 5.1 decisions. The planner must address all five gaps.

### GAP-1: Single job in cd-base.yml (violates D-01)

**Current:** `cd-base.yml` has one job named `deploy` that does everything: checkout, WIF auth, docker build+push, Pulumi setup, `pulumi up/preview`.

**Required (D-01):** Two jobs — `build` (checkout, WIF auth, docker build+push) and `deploy` (Pulumi install, `pulumi login`, `pulumi up/preview`). `deploy` depends on `build`.

**Impact:** Build step naming, `needs: [build]` in deploy job, job outputs to pass SHA from build to deploy.

### GAP-2: No Docker layer cache in build job (violates D-03)

**Current:** `cd-base.yml` runs `docker build` with no caching — just `docker build -t ...`.

**Required (D-03):** Docker layer cache via `actions/cache` or Docker Buildx GHA cache (`type=gha`).

**Recommendation (Claude's discretion):** Use `docker/setup-buildx-action` + `docker/build-push-action` with `cache-from: type=gha` and `cache-to: type=gha,mode=max`. This is the current Docker-recommended approach (GHA cache backend) and does not require a separate `actions/cache` step. [VERIFIED: docs.docker.com/build/ci/github-actions/cache/]

### GAP-3: Staging triggers on pull_request (violates D-05)

**Current:** `cd-staging.yml` triggers on BOTH `pull_request` (running `preview`) and `workflow_dispatch` (running `up`).

**Required (D-05):** `workflow_dispatch` ONLY — no PR preview, no auto-deploy. Remove the `pull_request` trigger and the `preview` conditional job entirely.

### GAP-4: No post-deploy health check on dev (violates D-04)

**Current:** `cd-dev.yml` / `cd-base.yml` has no health check after `pulumi up`.

**Required (D-04):** After the deploy job completes for `dev`, curl `https://dev.usevici.com/health`, fail if not 200 within 60s.

**Implementation pattern (Claude's discretion):**

```yaml
- name: Health check
  run: |
    for i in $(seq 1 12); do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://dev.usevici.com/health)
      if [ "$STATUS" = "200" ]; then
        echo "Health check passed (attempt $i)"
        exit 0
      fi
      echo "Attempt $i: got $STATUS, retrying in 5s..."
      sleep 5
    done
    echo "Health check failed after 60s"
    exit 1
```

12 retries × 5s = 60s max. [ASSUMED — interval/count are Claude's discretion per CONTEXT.md]

### GAP-5: Image tag and app:imageTag config key (violates D-07, D-08)

**Current:**
- `cd-base.yml` tags with both SHA and stack name: `docker push "${IMAGE}:${SHA}"` AND `docker push "${IMAGE}:${inputs.stack}"`
- `app.py` line 122: `image=pulumi.Output.concat(registry_url, "/vici:", ENV)` — uses ENV (e.g., "dev") as the image tag

**Required (D-07, D-08):**
- Tag with 7-char SHA only: `docker push "${IMAGE}:${SHA}"`
- Pass SHA to Pulumi via `--config app:imageTag=<sha>`
- `app.py` must read `imageTag` from Pulumi config, not `ENV`

**Code changes required:**

`infra/config.py` — add one line:
```python
IMAGE_TAG: str = cfg.get("imageTag") or ENV  # CI passes --config app:imageTag=<sha>; local fallback to ENV
```

`infra/components/app.py` — update import and line 122:
```python
from config import ENV, IMAGE_TAG  # add IMAGE_TAG
# line 122:
image=pulumi.Output.concat(registry_url, "/vici:", IMAGE_TAG),
```

**Pulumi CLI flag syntax (how to pass from workflow):**

Option A — `pulumi/actions@v6` `config-map` input:
```yaml
- uses: pulumi/actions@v6
  with:
    command: ${{ inputs.command }}
    stack-name: ${{ inputs.stack }}
    work-dir: infra
    cloud-url: gs://vici-app-pulumi-state-${{ inputs.stack }}
    config-map: '{"app:imageTag": {"value": "${{ needs.build.outputs.sha }}", "secret": false}}'
```
[VERIFIED: raw.githubusercontent.com/pulumi/actions/main/action.yml — `config-map` input exists]

Option B — inline CLI:
```bash
pulumi up --stack ${{ inputs.stack }} --config app:imageTag=${{ needs.build.outputs.sha }}
```

**Recommendation:** Use the `pulumi/actions@v6` `config-map` input (Option A) — it keeps all Pulumi invocation in the action rather than mixing raw CLI calls. However, the `config-map` format must use the fully-qualified key `app:imageTag` (namespace prefix required since it's in the `vici-infra` project config namespace, but Pulumi CLI uses the project name — verify the exact key name). [ASSUMED — key namespace behavior with config-map vs CLI flag needs testing]

**Alternative safe approach:** Run `pulumi up` directly via bash with the `--config` flag. The `cd-base.yml` `deploy` job can do:

```bash
cd infra
pulumi login gs://vici-app-pulumi-state-${{ inputs.stack }}
pulumi ${{ inputs.command }} --stack ${{ inputs.stack }} --config app:imageTag=${{ needs.build.outputs.sha }} --non-interactive
```

This avoids any uncertainty about `config-map` key format. This is the more robust pattern given the stack name in `Pulumi.yaml` is `vici-infra` and Pulumi config keys are namespaced as `vici-infra:imageTag`.

### GAP-6: Secret naming convention (D-09, D-11 vs current)

**Current:** Per-env secrets (`GCP_WIF_PROVIDER_DEV`, `GCP_WIF_PROVIDER_STAGING`, `GCP_WIF_PROVIDER_PROD`, `GCP_CI_SA_DEV`, `GCP_CI_SA_STAGING`, `GCP_CI_SA_PROD`).

**CONTEXT.md D-09/D-11:** Names `GCP_WIF_PROVIDER` and `GCP_CI_SA_EMAIL` (singular, not per-env).

**Research finding:** The per-env naming (`_DEV`, `_STAGING`, `_PROD`) is actually correct given each environment is a separate GCP project (`vici-app-dev`, `vici-app-staging`, `vici-app-prod`) with its own WIF pool provisioned in `components/cd.py` using `ENV` suffix (`github-actions-{ENV}`). The singular naming in D-09/D-11 was likely written assuming the ENV suffix would be handled by the workflow, but the per-env secrets approach is simpler and already implemented.

**Recommendation (Claude's discretion):** Keep the per-env secret naming (`GCP_WIF_PROVIDER_DEV`, etc.). This aligns with the existing codebase convention. The planner should note that `GCP_WIF_PROVIDER` in D-09 maps to `GCP_WIF_PROVIDER_DEV` (and `_STAGING`, `_PROD` variants) in practice.

**`PULUMI_CONFIG_PASSPHRASE` requirement:** Each stack config (`Pulumi.dev.yaml`, etc.) has `encryptionsalt` entries for encrypted values (temporal DB credentials). `PULUMI_CONFIG_PASSPHRASE` is required to decrypt these during `pulumi up`. This is correctly handled in the existing workflows and must be retained. [VERIFIED: infra/Pulumi.dev.yaml contains `encryptionsalt`]

## Architecture Patterns

### Recommended Workflow Structure

```
.github/workflows/
├── ci.yml           # EXISTING — do NOT modify (CD-05)
├── cd-base.yml      # Reusable; workflow_call; two jobs: build + deploy
├── cd-dev.yml       # Caller; push to main trigger; stack=dev command=up
├── cd-staging.yml   # Caller; workflow_dispatch only; stack=staging command=up
└── cd-prod.yml      # Caller; workflow_dispatch; stack=prod command=up; environment=prod
```

### Pattern 1: Reusable Workflow with Two Jobs (D-01)

**What:** `cd-base.yml` defines `build` and `deploy` as separate jobs. `build` emits SHA as a job output. `deploy` consumes SHA via `needs.build.outputs.sha`.

**When to use:** Any time you want CI to run Pulumi with an image built in the same workflow run.

```yaml
# Source: github.com/pulumi/actions (adapted)
on:
  workflow_call:
    inputs:
      stack:
        required: true
        type: string
      command:
        required: true
        type: string
      environment:
        required: false
        type: string
    secrets:
      WIF_PROVIDER:
        required: true
      WIF_SERVICE_ACCOUNT:
        required: true
      PULUMI_CONFIG_PASSPHRASE:
        required: true
    outputs:
      sha:
        description: "7-char git SHA used as image tag"
        value: ${{ jobs.build.outputs.sha }}

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    outputs:
      sha: ${{ steps.sha.outputs.sha }}
    steps:
      - uses: actions/checkout@v4
      - id: sha
        run: echo "sha=$(git rev-parse --short HEAD)" >> "$GITHUB_OUTPUT"
      - uses: google-github-actions/auth@v3
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}
      - uses: docker/setup-buildx-action@v3
      - name: Configure Docker for Artifact Registry
        run: gcloud auth configure-docker us-central1-docker.pkg.dev --quiet
      - uses: docker/build-push-action@v6
        if: inputs.command == 'up'
        with:
          context: .
          push: true
          tags: us-central1-docker.pkg.dev/${{ env.GCP_PROJECT }}/vici-images/vici:${{ steps.sha.outputs.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    runs-on: ubuntu-latest
    needs: [build]
    environment: ${{ inputs.environment }}
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/auth@v3
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}
      - uses: astral-sh/setup-uv@v5
        with:
          enable-cache: true
      - name: Install Pulumi dependencies
        run: cd infra && uv sync --frozen
      - uses: pulumi/actions@v6
        with:
          command: ${{ inputs.command }}
          stack-name: ${{ inputs.stack }}
          work-dir: infra
          cloud-url: gs://vici-app-pulumi-state-${{ inputs.stack }}
        env:
          PULUMI_CONFIG_PASSPHRASE: ${{ secrets.PULUMI_CONFIG_PASSPHRASE }}
          # imageTag passed as CLI arg via config-map or env — see GAP-5 discussion
```

**Critical: `id-token: write` in reusable workflows.** Both `build` and `deploy` jobs need `id-token: write` at the job level. The caller workflow (`cd-dev.yml`, etc.) also needs `permissions: id-token: write`. The reusable workflow permissions cannot be elevated beyond what the caller grants. [VERIFIED: github.blog/changelog/2023-06-15-github-actions-securing-openid-connect-oidc-token-permissions-in-reusable-workflows/]

### Pattern 2: Job Outputs to Pass SHA Between Jobs

**What:** `build` job captures SHA in `steps.sha.outputs.sha`, exposes as `jobs.build.outputs.sha`. `deploy` job references via `${{ needs.build.outputs.sha }}`.

```yaml
# In build job:
- id: sha
  run: echo "sha=$(git rev-parse --short HEAD)" >> "$GITHUB_OUTPUT"
outputs:
  sha: ${{ steps.sha.outputs.sha }}

# In deploy job:
needs: [build]
# Reference:
${{ needs.build.outputs.sha }}
```

[VERIFIED: docs.github.com/en/actions/writing-workflows/choosing-what-your-workflow-does/passing-information-between-jobs]

### Pattern 3: GCP Project ID in Build Job

**Problem:** The `docker build-push-action` tag line needs `$GCP_PROJECT` but WIF auth does not automatically set `GCLOUD_PROJECT` env var.

**Solution:** After `google-github-actions/auth@v3`, call `gcloud config get-value project` to extract PROJECT_ID, or set it from a GitHub Actions variable. The current `cd-base.yml` uses shell expansion `$(gcloud config get-value project)` — this pattern works after WIF auth because `gcloud` picks up the configured project. However, the WIF auth step needs `--project` or the project must be set in WIF provider.

**Alternative:** Add a `project` workflow input (or per-env GitHub Actions variable), pass as `GCP_PROJECT_ID` env var to the build step. This is cleaner and avoids a `gcloud` call.

[ASSUMED — need to verify gcloud project config is set correctly after google-github-actions/auth@v3 with WIF]

### Pattern 4: Pulumi Config Key for imageTag

**The Pulumi config key namespace:** In `Pulumi.dev.yaml`, keys are `vici-infra:env`, `vici-infra:cluster_name`, etc. The `cfg = pulumi.Config()` in `config.py` automatically scopes to the `vici-infra` project namespace. Therefore `cfg.get("imageTag")` reads `vici-infra:imageTag`.

**CLI flag to set it:** `pulumi up --config vici-infra:imageTag=a1b2c3d` (not `app:imageTag` as stated in D-08). The CONTEXT.md D-08 uses `app:imageTag` which is incorrect for this Pulumi project; the correct namespace is `vici-infra:imageTag`.

[VERIFIED: infra/Pulumi.dev.yaml — all keys prefixed with `vici-infra:`; infra/config.py — `cfg = pulumi.Config()` with no explicit namespace (uses project name)]

**`config-map` format for `pulumi/actions@v6`:**
```yaml
config-map: '{"vici-infra:imageTag": {"value": "${{ needs.build.outputs.sha }}", "secret": false}}'
```
[VERIFIED: raw.githubusercontent.com/pulumi/actions/main/action.yml — `config-map` input exists]

### Anti-Patterns to Avoid

- **Do NOT add `pull_request` trigger to `cd-staging.yml`:** D-05 explicitly removed this.
- **Do NOT use `secrets: inherit` in caller workflows:** Explicitly pass secrets to maintain auditability and avoid accidentally leaking org-level secrets.
- **Do NOT store GCP SA keys as GitHub secrets:** WIF is the locked auth mechanism (CD-04).
- **Do NOT modify `ci.yml`:** CD-05 prohibits this. The test job remains SQLite-only.
- **Do NOT push image with env tag (e.g., `:dev`):** D-07 specifies SHA-only tags. Mutable env tags cause non-reproducible deploys.

## Common Pitfalls

### Pitfall 1: id-token Permission Missing from Caller Workflow
**What goes wrong:** Reusable workflow requests `id-token: write` but caller only grants `id-token: none` (default). GitHub rejects with "The workflow is requesting 'id-token: write', but is only allowed 'id-token: none'."
**Why it happens:** Default repository permissions for `id-token` are `none`. Must be explicitly granted.
**How to avoid:** Add `permissions: id-token: write` to the caller workflow (`cd-dev.yml`, etc.) at the top level AND within each job that calls the reusable workflow.
**Warning signs:** Workflow fails immediately at auth step with permission error.
[VERIFIED: github.com/orgs/community/discussions/121112]

### Pitfall 2: WIF Binding Scope Too Narrow / Too Broad
**What goes wrong:** WIF IAM binding in `cd.py` uses `attribute.repository/ahcarpenter/vici` — only tokens from the `ahcarpenter/vici` repo can impersonate `ci_push_sa`. This is correct but means forks and other repos cannot push.
**Why it matters:** If the repo is renamed or transferred, the WIF binding breaks. Currently `GITHUB_ORG = ahcarpenter` per `config.py` and `Pulumi.dev.yaml`.
**How to avoid:** No action needed — this is correctly scoped. Document that changing the repo name or org requires `pulumi up` to update the binding.
[VERIFIED: infra/components/cd.py line 62-76]

### Pitfall 3: Docker Build Context vs. Pulumi Work Dir
**What goes wrong:** `cd-base.yml` runs `docker build` from the repo root (correct) but Pulumi runs from `infra/` subdirectory. If `working-directory` is set globally, docker build breaks.
**How to avoid:** Pulumi step uses `work-dir: infra` input to `pulumi/actions@v6`. Docker build step runs from repo root (default). Do not set a global `defaults.run.working-directory`.

### Pitfall 4: PULUMI_CONFIG_PASSPHRASE and Encrypted Stack Secrets
**What goes wrong:** `pulumi up` fails with "passphrase is required" because the stack configs contain encrypted values (`encryptionsalt` in `Pulumi.dev.yaml`, etc.).
**Why it happens:** Each stack has `vici-infra:temporal_db_user` and `vici-infra:temporal_db_password` as encrypted secrets requiring the passphrase.
**How to avoid:** Always pass `PULUMI_CONFIG_PASSPHRASE` as a secret to `cd-base.yml`. This is correctly implemented in the existing workflows but NOT mentioned in CONTEXT.md D-09/D-11. Retain `PULUMI_CONFIG_PASSPHRASE` as a required secret in `cd-base.yml`'s `on.workflow_call.secrets`.
[VERIFIED: infra/Pulumi.dev.yaml — contains `encryptionsalt`]

### Pitfall 5: Pulumi Config Key Namespace Mismatch
**What goes wrong:** CONTEXT.md D-08 says `--config app:imageTag=<sha>` but the correct namespace for this project is `vici-infra:imageTag` (project name from `Pulumi.yaml`).
**Why it happens:** D-08 was written with a generic example; the actual project name is `vici-infra`.
**How to avoid:** Use `--config vici-infra:imageTag=<sha>` in the workflow, or use `pulumi/actions@v6` `config-map` with key `"vici-infra:imageTag"`.
[VERIFIED: infra/Pulumi.yaml — `name: vici-infra`; infra/config.py — `cfg = pulumi.Config()` uses project name namespace]

### Pitfall 6: GCP Project Varies Per Environment
**What goes wrong:** Each environment uses a different GCP project (`vici-app-dev`, `vici-app-staging`, `vici-app-prod`), each with its own Artifact Registry at a different path. The image URI in `cd-base.yml` must use the correct `$PROJECT_ID` per environment.
**Why it happens:** The existing `cd-base.yml` shell script calls `gcloud config get-value project` after WIF auth — this works IF WIF auth sets the project correctly. However, `google-github-actions/auth@v3` does not automatically set the gcloud project from the WIF provider's project — you must pass the project ID explicitly.
**How to avoid:** Add a `gcp_project` input to `cd-base.yml`, pass per-env from caller workflows (e.g., `gcp_project: vici-app-dev`). Use `${{ inputs.gcp_project }}` in the docker tag.
[ASSUMED — the WIF provider is in the same GCP project; auth action behavior with `gcloud config get-value project` needs verification. The existing code uses this pattern but it may fail if project is not configured in WIF provider token.]

```yaml
      sha: ${{ steps.sha.outputs.sha }}
    steps:
      - uses: actions/checkout@v4
      - id: sha
        run: echo "sha=$(git rev-parse --short=7 HEAD)" >> "$GITHUB_OUTPUT"
      - uses: google-github-actions/auth@v3
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}
      - uses: docker/setup-buildx-action@v3
      - name: Configure Docker for Artifact Registry
        run: gcloud auth configure-docker us-central1-docker.pkg.dev --quiet
      - uses: docker/build-push-action@v6
        if: inputs.command == 'up'
        with:
          context: .
          push: true
          tags: us-central1-docker.pkg.dev/${{ inputs.gcp_project }}/vici-images/vici:${{ steps.sha.outputs.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    runs-on: ubuntu-latest
    needs: [build]
    environment: ${{ inputs.environment }}
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/auth@v3
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}
      - uses: astral-sh/setup-uv@v5
        with:
          enable-cache: true
      - name: Install Pulumi dependencies
        run: cd infra && uv sync --frozen
      - uses: pulumi/actions@v6
        with:
          command: ${{ inputs.command }}
          stack-name: ${{ inputs.stack }}
          work-dir: infra
          cloud-url: gs://vici-app-pulumi-state-${{ inputs.stack }}
          config-map: '{"vici-infra:imageTag": {"value": "${{ needs.build.outputs.sha }}", "secret": false}}'
        env:
          PULUMI_CONFIG_PASSPHRASE: ${{ secrets.PULUMI_CONFIG_PASSPHRASE }}
      - name: Health check (dev only)
        if: inputs.stack == 'dev' && inputs.command == 'up'
        run: |
          for i in $(seq 1 12); do
            STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://dev.usevici.com/health)
            if [ "$STATUS" = "200" ]; then
              echo "Health check passed (attempt $i)"
              exit 0
            fi
            echo "Attempt $i: got $STATUS, retrying in 5s..."
            sleep 5
          done
          echo "Health check failed after 60s"
          exit 1
```

## Open Questions

1. **config-map key format for pulumi/actions@v6**
   - What we know: The input `config-map` exists in `action.yml`. Format is `{key: {value: ..., secret: ...}}`.
   - What's unclear: Does the key require the project namespace prefix (`vici-infra:imageTag`) or just the bare name (`imageTag`)? Official docs show bare key example `{name: {value: my-pet}}`.
   - Recommendation: Use bare key `imageTag` in `config-map` first (matches Pulumi's `cfg.get("imageTag")` which auto-namespaces). If that fails, try `vici-infra:imageTag`. Alternatively, skip `config-map` entirely and call `pulumi up --config vici-infra:imageTag=<sha>` via bash for unambiguous behavior.

2. **GCP Project from WIF auth context**
   - What we know: Existing `cd-base.yml` uses `$(gcloud config get-value project)` after auth. `google-github-actions/auth@v3` sets up gcloud credentials.
   - What's unclear: Does auth action set the gcloud `project` config property automatically from the WIF provider's project?
   - Recommendation: Eliminate the question by adding a `gcp_project` input to `cd-base.yml` (already in the recommended target state above). This makes the project explicit and avoids runtime discovery.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | WIF OIDC — `google-github-actions/auth@v3`; no static credentials |
| V3 Session Management | no | CI jobs are ephemeral; no sessions |
| V4 Access Control | yes | WIF IAM binding scoped to `ahcarpenter/vici` repo only (not all repos) |
| V5 Input Validation | yes | `workflow_call` inputs typed as `string` / `boolean`; no injection vectors in workflow YAML |
| V6 Cryptography | yes | `PULUMI_CONFIG_PASSPHRASE` for stack secret decryption — stored as GitHub encrypted secret |

### Known Threat Patterns for GitHub Actions + GCP WIF

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Stolen OIDC token used from outside repo | Spoofing | `attribute_condition` in WIF provider restricts to `ahcarpenter/vici` repo only |
| Malicious PR triggering deploy | Tampering | Staging and prod use `workflow_dispatch` only; dev auto-deploys on merge to main (not PR) |
| Secret exfiltration from workflow logs | Information Disclosure | `PULUMI_CONFIG_PASSPHRASE` and SA email passed as secrets (masked in logs); `suppress-outputs: true` on Pulumi action optional |
| Image tampered between build and deploy | Tampering | SHA-pinned tags (D-07) — deploy job references same SHA built in build job |
| Unauthorized prod deploy | Elevation of Privilege | GitHub Environment `prod` requires manual approval from designated reviewers |
