# Phase 05.1: Code Review Report

**Reviewed:** 2026-04-11
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

The Phase 05.1 CD rewrite is structurally sound. The `IMAGE_TAG` Pulumi config key is
wired end-to-end (config.py -> app.py via `pulumi.Output.concat`), WIF auth is configured
on both build and deploy jobs with the required `id-token: write` permission, the reusable
workflow cleanly separates build from deploy, and the `"on":` key is quoted to dodge the
YAML 1.1 `on -> True` boolean coercion that breaks `yaml.safe_load` consumers.

Three issues merit attention before this lands:

1. **Prod approval gate is bypassed for image push.** The `environment: prod` gate is
   only attached to the `deploy` job. The `build` job runs first, authenticates to prod
   WIF, and pushes an arbitrary image to `vici-app-prod` Artifact Registry *before* any
   human approves. For `workflow_dispatch` on a non-default branch this means an engineer
   with workflow_dispatch permission can land arbitrary bytes in the prod registry
   without review. This defeats the stated purpose of the approval gate.

2. **No concurrency control.** Two pushes to `main` in quick succession will race two
   Pulumi deploys against the same dev stack, producing Pulumi state-lock conflicts or
   interleaved updates. A `concurrency:` group per stack is the standard fix.

3. **Caller-level permissions blocks are dead code.** `cd-dev.yml`, `cd-staging.yml`,
   and `cd-prod.yml` each declare `permissions: { contents: read, id-token: write }`
   at the workflow level, but per GitHub Actions semantics, permissions for a reusable
   workflow's jobs come from the *called* workflow, not the caller. These blocks have
   no effect and create a false sense of security when auditing.

The static tests mostly pin the right contracts, but several are weak enough that a
regression (e.g. hardcoded `imageTag`, missing `needs.build.outputs.sha` wiring) would
slip through. Callouts below.

## Warnings

### WR-01: Prod approval gate does not cover image build/push

**File:** `.github/workflows/cd-base.yml:31-61`, `.github/workflows/cd-prod.yml:11-17`
**Issue:** `cd-base.yml` assigns `environment: ${{ inputs.environment }}` only to the
`deploy` job (line 66). The `build` job authenticates to GCP via WIF and pushes
`us-central1-docker.pkg.dev/vici-app-prod/vici-images/vici:<sha>` before any approval
gate fires. The `prod` GitHub Environment's required-reviewers protection only blocks
`deploy` from starting; it does not block `build` from pushing an image into the prod
Artifact Registry. Anyone who can trigger `workflow_dispatch` on `cd-prod.yml` can push
an arbitrary image to prod without human approval. The deploy-time gate then cannot
distinguish a reviewed SHA from an unreviewed one — it only sees whatever tag the build
job produced.
**Fix:** Attach the environment gate to the build job as well. Either:

```yaml
# cd-base.yml build job
build:
  runs-on: ubuntu-latest
  environment: ${{ inputs.environment }}
  permissions:
    contents: read
    id-token: write
  # ...
```

Or, cleaner, split the prod flow so the build runs from a trusted branch snapshot and
only the deploy stage is gated. Given the current two-job shape, the simpler fix is to
add `environment: ${{ inputs.environment }}` to `build` so both jobs observe the same
approval gate.

### WR-02: No concurrency control on dev auto-deploy

**File:** `.github/workflows/cd-dev.yml:1-22`, `.github/workflows/cd-base.yml:1-109`
**Issue:** None of the four CD workflows declare a `concurrency:` group. Two pushes to
`main` within a few minutes will launch two concurrent `cd-dev` runs, both calling into
`cd-base.yml`, both issuing `pulumi up` against the same GCS-backed dev stack. Pulumi
takes a per-stack state lock, so the second run will either fail loudly on lock
contention or, worse, interleave resource updates if the first run finishes between the
second run's plan and apply. This is especially likely during a merge storm on main.
**Fix:** Add a concurrency group keyed on stack:

```yaml
# cd-dev.yml (and mirror in cd-staging.yml, cd-prod.yml)
concurrency:
  group: cd-${{ github.workflow }}
  cancel-in-progress: false
```

Use `cancel-in-progress: false` on prod/staging so approved deploys are never abandoned
mid-apply; `true` is acceptable for dev where the latest main should win.

### WR-03: Health check step runs on the wrong job boundary for failure isolation

**File:** `.github/workflows/cd-base.yml:95-108`
**Issue:** The health check is the last step of the `deploy` job. If `pulumi up`
succeeds but the new pods never go healthy (CrashLoopBackOff, failed readiness, wrong
secret), the job fails *after* Pulumi has already recorded a successful update in
state. There is no automatic rollback, and the only signal in the Pulumi state is the
new image tag. Additionally, the loop runs `sleep 5` twelve times (60s total), which
may be too short if the deployment's `initial_delay_seconds` for readiness is 15s and
the new pod still has to pull the image. With `docker/build-push-action` pushing to a
private registry and `imagePullPolicy: Always` on the container (app.py:125), the pull
alone can consume most of that window.
**Fix:** Two improvements:

1. Extend the budget to at least 3 minutes (e.g. 36 iterations of 5s, or 18 of 10s) to
   cover image pull + rolling update + readiness delay.
2. On failure, log pod status to aid triage:

```yaml
- name: Health check
  if: inputs.stack == 'dev' && inputs.command == 'up'
  run: |
    for i in $(seq 1 36); do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://dev.usevici.com/health)
      if [ "$STATUS" = "200" ]; then
        echo "Health check passed (attempt $i)"
        exit 0
      fi
      echo "Attempt $i: got $STATUS, retrying in 5s..."
      sleep 5
    done
    echo "Health check failed after 180s"
    exit 1
```

Consider whether rollback-on-failure (another `pulumi up` with the previous tag) is in
scope; if so, that belongs in a separate step after this one.

## Info

### IN-01: Caller workflows declare dead `permissions:` blocks

**File:** `.github/workflows/cd-dev.yml:7-9`, `.github/workflows/cd-staging.yml:6-8`, `.github/workflows/cd-prod.yml:6-8`
**Issue:** The top-level `permissions: { contents: read, id-token: write }` in the three
caller workflows has no effect on the reusable workflow's jobs. When `cd-dev.yml` calls
`cd-base.yml`, the jobs run in `cd-base.yml`'s context and consult *its* `permissions`
declarations (which are correctly set on each job). The caller's block is dead config.
It is not harmful (the caller workflow has no jobs of its own), but it misleads anyone
auditing permissions since it suggests the caller is granting id-token access to the
called workflow — it is not.
**Fix:** Remove the top-level `permissions:` block from all three caller workflows, or
add a comment explaining it is a no-op retained for clarity:

```yaml
# cd-dev.yml — remove lines 7-9 entirely
name: CD Dev

"on":
  push:
    branches: [main]

jobs:
  deploy-dev:
    uses: ./.github/workflows/cd-base.yml
    # ...
```

### IN-02: `config-map` JSON uses string interpolation instead of `env:`

**File:** `.github/workflows/cd-base.yml:91`
**Issue:** The `config-map` JSON is built by interpolating `${{ needs.build.outputs.sha }}`
directly into the YAML string:

```yaml
config-map: '{"vici-infra:imageTag": {"value": "${{ needs.build.outputs.sha }}", "secret": false}}'
```

Today this is safe because `sha` is the output of `git rev-parse --short=7` (always
`[0-9a-f]{7}`, no quote or brace characters). But the pattern is fragile: any future
refactor that broadens the source (e.g. a `GITHUB_REF_NAME`-based tag for PR previews)
would silently open a YAML/JSON injection path. This is the same class of problem that
GitHub's "unsafe expressions in run blocks" guidance warns about.
**Fix:** Pass the value through an environment variable so the interpolation happens in
the shell, not in the YAML template:

```yaml
- uses: pulumi/actions@v6
  env:
    IMAGE_TAG: ${{ needs.build.outputs.sha }}
    PULUMI_CONFIG_PASSPHRASE: ${{ secrets.PULUMI_CONFIG_PASSPHRASE }}
  with:
    command: ${{ inputs.command }}
    stack-name: ${{ inputs.stack }}
    work-dir: infra
    cloud-url: gs://vici-app-pulumi-state-${{ inputs.stack }}
    config-map: |
      vici-infra:imageTag:
        value: ${IMAGE_TAG}
        secret: false
```

Or, even better, rely on Pulumi's native env-var config: set `PULUMI_CONFIG_IMAGETAG` in
`env:` and drop `config-map` entirely (Pulumi reads `PULUMI_CONFIG_<UPPERCASE_KEY>` for
non-namespaced keys; for namespaced keys use `pulumi config set` in a prior step).

### IN-03: `docker/build-push-action` is skipped for preview but deploy still passes image tag

**File:** `.github/workflows/cd-base.yml:54-61, 85-93`
**Issue:** `docker/build-push-action@v6` has `if: inputs.command == 'up'` (line 55), so
on `command: preview` no image is built or pushed. However, the deploy job unconditionally
passes `vici-infra:imageTag=<sha>` via `config-map` (line 91). For a preview of a brand
new SHA, Pulumi will compute a diff against a tag that does not yet exist in Artifact
Registry. Preview will succeed (it only resolves the string), but the diff will claim a
new container image is ready when it is not. This is a correctness-of-diff concern, not
a runtime bug — but anyone reviewing a preview to decide whether to approve would see a
reassuring "no image change needed" / "image change pending" output that does not reflect
registry reality.
**Fix:** Either (a) keep current behavior and document that `command: preview` diffs are
advisory-only for image changes, or (b) run the build+push unconditionally and let
Pulumi decide whether to update the deployment, or (c) on `preview`, derive the image
tag from the currently-deployed stack rather than from HEAD SHA.

### IN-04: `test_cd_base_uses_pulumi_config_map_for_image_tag` does not pin the value source

**File:** `tests/infra/test_cd_workflows_static.py:318-331`
**Issue:** The test only asserts that the string `"imageTag"` appears somewhere in the
`config-map` value. A regression that sets `imageTag` to a hardcoded literal (e.g.
`"dev"`) or to the wrong input (e.g. `inputs.stack`) would still pass. Given that CD-06
is the entire reason this phase exists — wiring the build SHA into the deploy's Pulumi
config — the test should pin that wiring.
**Fix:**

```python
def test_cd_base_uses_pulumi_config_map_for_image_tag(self) -> None:
    steps = _all_steps(self.workflow, "deploy")
    pulumi_steps = [
        s for s in steps if "pulumi/actions" in (s.get("uses", "") or "")
    ]
    assert pulumi_steps, "cd-base.yml deploy job must have a pulumi/actions step"
    step_with = pulumi_steps[0].get("with", {})
    config_map = str(step_with.get("config-map", "") or "")
    assert "vici-infra:imageTag" in config_map, (
        f"pulumi/actions must set vici-infra:imageTag, got: {config_map!r}"
    )
    assert "needs.build.outputs.sha" in config_map, (
        f"pulumi/actions imageTag must come from build job's sha output, got: {config_map!r}"
    )
```

### IN-05: Static tests do not verify secret passthrough from callers to reusable workflow

**File:** `tests/infra/test_cd_workflows_static.py` (entire file)
**Issue:** The tests verify that `cd-base.yml` declares `WIF_PROVIDER`,
`WIF_SERVICE_ACCOUNT`, and `PULUMI_CONFIG_PASSPHRASE` as required secrets only indirectly
(via the "uses google-github-actions/auth@v3" check). There is no test that each caller
(`cd-dev.yml`, `cd-staging.yml`, `cd-prod.yml`) actually wires those three secrets in its
`secrets:` block. A caller that forgot to pass `PULUMI_CONFIG_PASSPHRASE` would fail at
runtime with an opaque Pulumi decryption error.
**Fix:** Add a parameterized test:

```python
@pytest.mark.parametrize("filename", ["cd-dev.yml", "cd-staging.yml", "cd-prod.yml"])
def test_caller_passes_required_secrets(filename: str) -> None:
    workflow = _load_workflow(filename)
    job = next(iter(workflow.get("jobs", {}).values()))
    secrets = job.get("secrets", {}) or {}
    for required in ("WIF_PROVIDER", "WIF_SERVICE_ACCOUNT", "PULUMI_CONFIG_PASSPHRASE"):
        assert required in secrets, (
            f"{filename} must pass {required} through to cd-base.yml, "
            f"got secrets: {list(secrets.keys())}"
        )
```

### IN-06: `IMAGE_TAG` fallback to `ENV` masks misconfiguration in CI

**File:** `infra/config.py:13`
**Issue:** `IMAGE_TAG: str = cfg.get("imageTag") or ENV` is a reasonable local-dev
convenience — it lets engineers run `pulumi up` without needing to pass `--config
vici-infra:imageTag=<sha>`. But in CI, if the `config-map` passthrough ever breaks
(typo in namespace, wrong key casing, Pulumi action version change), this fallback
will silently deploy `<registry>/vici:dev` — an image that may or may not exist,
and certainly is not the freshly-built SHA. The deploy would succeed and either pull
a stale image or fail at pod pull time, far from the root cause. The test in
`test_cd_static.py::test_config_image_tag_has_env_fallback` explicitly pins this
string pattern, so breaking the fallback intentionally in CI would require test
updates.
**Fix:** Detect CI explicitly and fail fast when `imageTag` is unset there. For example:

```python
import os

_IS_CI = os.environ.get("CI") == "true" or os.environ.get("GITHUB_ACTIONS") == "true"
_image_tag_from_config = cfg.get("imageTag")
if _image_tag_from_config is None and _IS_CI:
    raise pulumi.RunError(
        "vici-infra:imageTag is required in CI — check cd-base.yml config-map wiring"
    )
IMAGE_TAG: str = _image_tag_from_config or ENV
```

This preserves local-dev ergonomics while making CI misconfiguration loud instead of
silent. Alternative: drop the fallback entirely and require `imageTag` to be set in the
stack YAML with a sentinel like `local`, then let CI override.
