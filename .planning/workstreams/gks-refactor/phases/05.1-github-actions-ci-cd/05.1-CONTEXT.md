# Phase 5.1: GitHub Actions CI/CD - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Automated CI/CD pipeline via GitHub Actions with Workload Identity Federation auth. Pushing to main triggers build and deploy to dev; staging and prod are manual dispatch with prod requiring GitHub Environment approval. No application code changes — infrastructure and workflow files only.

</domain>

<decisions>
## Implementation Decisions

### Workflow structure
- **D-01:** Reusable `cd-base.yml` with `workflow_call` — two jobs: **build** (checkout, WIF auth, docker build+push) and **deploy** (Pulumi install, login to GCS, pulumi up/preview). Callers pass stack name and command.
- **D-02:** `cd-base.yml` handles all Pulumi setup internally (install, `pulumi login` to GCS backend). Callers only pass stack name and Pulumi command (`up` or `preview`).
- **D-03:** Build job runs on every CD trigger — no registry existence check. Docker layer cache via `actions/cache` for speed.

### Deploy gating
- **D-04:** Dev: auto-deploy on push to `main` via `cd-dev.yml`. Includes post-deploy health check — curl the dev Ingress `/health` endpoint, fail if not 200 within 60s.
- **D-05:** Staging: manual `workflow_dispatch` only via `cd-staging.yml`. No PR preview, no auto-deploy.
- **D-06:** Prod: manual `workflow_dispatch` via `cd-prod.yml` with GitHub Environment (`prod`) requiring reviewers. Native GitHub approval gate.

### Image tagging
- **D-07:** Docker images tagged with 7-char git SHA (e.g., `us-central1-docker.pkg.dev/PROJECT/vici/app:a1b2c3d`). Immutable, traceable to exact commit.
- **D-08:** Image tag passed to Pulumi via CLI flag: `--config app:imageTag=<sha>`. No stack file changes on deploy.

### Secrets & config
- **D-09:** WIF provider full name stored as GitHub repo secret (`GCP_WIF_PROVIDER`). Set once after first `pulumi up`. Workflows reference `secrets.GCP_WIF_PROVIDER`.
- **D-10:** No `PULUMI_ACCESS_TOKEN` needed — Pulumi state is in GCS backend, WIF auth covers GCS access.
- **D-11:** Required GitHub repo secrets: `GCP_WIF_PROVIDER`, `GCP_CI_SA_EMAIL` (CI service account email for WIF).

### Claude's Discretion
- Exact GitHub Actions step naming and ordering within jobs
- Docker build context and caching strategy details
- Pulumi install method (action vs curl)
- Health check retry logic (interval, max attempts)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `components/cd.py`: WIF pool + provider + CI SA IAM bindings already implemented. Exports `wif_pool`, `wif_provider`.
- `components/identity.py`: `ci_push_sa` service account with Artifact Registry write access.
- `components/registry.py`: `registry_url` Pulumi export — use in workflow for image push target.
- `config.py`: `GITHUB_ORG` already required in stack config — used for WIF attribute_condition.

### Established Patterns
- All Pulumi components follow the same pattern: module-level resource creation with `pulumi.export()` at bottom.
- Stack configs (`Pulumi.dev.yaml`, `Pulumi.staging.yaml`, `Pulumi.prod.yaml`) hold env-specific values.
- `config.py` reads stack config via `pulumi.Config()`.

### Integration Points
- `__main__.py` already imports `from components.cd import wif_pool, wif_provider` — no wiring changes needed.
- `config.py` needs `app:imageTag` config key support for the CLI flag approach (D-08).
- GitHub Actions workflows go in `.github/workflows/` alongside existing `ci.yml`.

</code_context>
