Rewrite `.github/workflows/cd-base.yml` from scratch. The complete target content:

```yaml
name: CD Base

on:
  workflow_call:
    inputs:
      stack:
        required: true
        type: string
      command:
        required: true
        type: string
      environment:
        required: false
        type: string
      gcp_project:
        required: true
        type: string
    secrets:
      WIF_PROVIDER:
        required: true
      WIF_SERVICE_ACCOUNT:
        required: true
      PULUMI_CONFIG_PASSPHRASE:
        required: true
    outputs:
      sha:
        description: "7-char git SHA used as image tag"
        value: ${{ jobs.build.outputs.sha }}

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    outputs:
      sha: ${{ steps.sha.outputs.sha }}
    steps:
      - uses: actions/checkout@v4

      - id: sha
        run: echo "sha=$(git rev-parse --short=7 HEAD)" >> "$GITHUB_OUTPUT"

      - uses: google-github-actions/auth@v3
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}

      - uses: docker/setup-buildx-action@v3

      - name: Configure Docker for Artifact Registry
        run: gcloud auth configure-docker us-central1-docker.pkg.dev --quiet

      - uses: docker/build-push-action@v6
        if: inputs.command == 'up'
        with:
          context: .
          push: true
          tags: us-central1-docker.pkg.dev/${{ inputs.gcp_project }}/vici-images/vici:${{ steps.sha.outputs.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    runs-on: ubuntu-latest
    needs: [build]
    environment: ${{ inputs.environment }}
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4

      - uses: google-github-actions/auth@v3
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}

      - uses: astral-sh/setup-uv@v5
        with:
          enable-cache: true

      - name: Install Pulumi dependencies
        run: cd infra && uv sync --frozen

      - uses: pulumi/actions@v6
        with:
          command: ${{ inputs.command }}
          stack-name: ${{ inputs.stack }}
          work-dir: infra
          cloud-url: gs://vici-app-pulumi-state-${{ inputs.stack }}
          config-map: '{"vici-infra:imageTag": {"value": "${{ needs.build.outputs.sha }}", "secret": false}}'
        env:
          PULUMI_CONFIG_PASSPHRASE: ${{ secrets.PULUMI_CONFIG_PASSPHRASE }}

      - name: Health check
        if: inputs.stack == 'dev' && inputs.command == 'up'
        run: |
          for i in $(seq 1 12); do
            STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://dev.usevici.com/health)
            if [ "$STATUS" = "200" ]; then
              echo "Health check passed (attempt $i)"
              exit 0
            fi
            echo "Attempt $i: got $STATUS, retrying in 5s..."
            sleep 5
          done
          echo "Health check failed after 60s"
          exit 1
```

Key design decisions implemented:
- Per D-01: Two jobs `build` and `deploy`. Deploy has `needs: [build]`.
- Per D-02: Deploy job handles Pulumi install (uv sync) and login (cloud-url) internally.
- Per D-03: Docker layer cache via `docker/build-push-action@v6` with `cache-from: type=gha` and `cache-to: type=gha,mode=max`. Uses `docker/setup-buildx-action@v3` for Buildx driver.
- Per D-04: Health check step in deploy job runs only for `dev` + `up` (12 retries x 5s = 60s max).
- Per D-07: Only SHA tag — no env-name tag. Uses `--short=7` for exactly 7 chars.
- Per D-08: `config-map` uses `vici-infra:imageTag` key. NOTE: D-08 specifies `app:imageTag`, but the Pulumi project name is `vici-infra` (per `infra/Pulumi.yaml`), so the correct namespace is `vici-infra:imageTag`. This is an acknowledged deviation from D-08 wording, not from D-08 intent — Pulumi namespaces config keys by project name.
- Per D-10: No `PULUMI_ACCESS_TOKEN`. State access via GCS `cloud-url`.
- Per Research Pitfall 1: Both jobs have `permissions: id-token: write`.
- Per Research Pitfall 4: `PULUMI_CONFIG_PASSPHRASE` retained as required secret.
- Per Research Pitfall 6: `gcp_project` input eliminates runtime project discovery.
- Build job conditionally pushes only when `command == 'up'` (skips push for `preview`).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub OIDC -> GCP WIF | GitHub Actions runner exchanges OIDC token for GCP credentials via WIF provider |
| GitHub Secrets -> Workflow | Encrypted secrets injected into workflow runner environment |
| Docker image -> Artifact Registry | Built image pushed to GCP Artifact Registry |
| Pulumi CLI -> GCS state bucket | Pulumi reads/writes state from GCS using WIF credentials |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5.1-02 | Spoofing | WIF OIDC token | mitigate | `attribute_condition` in `cd.py` restricts WIF to `ahcarpenter/vici` repo only; no other repo can impersonate the CI SA |
| T-5.1-03 | Tampering | Malicious PR triggering deploy | mitigate | Dev deploys only on merge to `main` (not PR); staging/prod are `workflow_dispatch` only (D-05, D-06) |
| T-5.1-04 | Information Disclosure | Secret exfiltration from logs | mitigate | `PULUMI_CONFIG_PASSPHRASE` and SA email passed as GitHub encrypted secrets (masked in logs); no secrets in workflow file content |
| T-5.1-05 | Tampering | Image tampering between build and deploy | mitigate | SHA-pinned tags (D-07) — deploy job references exact SHA from build job output; no mutable tags |
| T-5.1-06 | Elevation of Privilege | Unauthorized prod deploy | mitigate | GitHub Environment `prod` requires reviewer approval before deploy job runs (CD-03, D-06) |
| T-5.1-07 | Information Disclosure | GCP project ID in workflow files | accept | Project IDs are not secrets; they appear in GCP console URLs and are required for Artifact Registry paths |
</threat_model>
