## Gray Areas Identified (Codebase Analysis Results)

These are ready to present to the user. Do NOT re-analyze the codebase — this analysis is complete.

### 1. Pulumi `protect=True` scope
- **Current state**: `cluster.py` has GCP-level `deletion_protection=True` but NO Pulumi-level `protect=True` on any resource. Cloud SQL instances, Artifact Registry, GCS state buckets have no protection from `pulumi destroy`.
- **Decision needed**: Which resources should have `protect=True`? All stateful resources (Cloud SQL instances + cluster)? Or only prod? ROADMAP.md success criterion 1 says "explicit `protect` flags on stateful resources."

### 2. Network policies
- **Current state**: Zero NetworkPolicy resources anywhere in the codebase.
- **Decision needed**: How strict? Namespace-level deny-all + allow-listed ports only? Which namespaces (all 6: vici, temporal, observability, cert-manager, external-secrets)? ROADMAP.md success criterion 3 says "restrict inter-namespace traffic to declared ports only."

### 3. Pod disruption budgets
- **Current state**: No PodDisruptionBudget resources. Autopilot manages node availability.
- **Decision needed**: PDBs are in success criterion 2, but single-node workloads (OpenSearch, Temporal in dev) make `minAvailable: 1` essentially a no-op. Should PDBs be env-conditional? Or skip for single-node workloads and document why?

### 4. Temporal DB credentials in Helm values
- **Current state**: `_TEMPORAL_DB_USER` and `_TEMPORAL_DB_PASS` are Pulumi stack secrets (encrypted at rest in `Pulumi.*.yaml`) but are injected as **plaintext** into the Helm values dict. They're visible in Pulumi state and Helm release history.
- **Decision needed**: Keep as-is (Pulumi stack secrets provide sufficient protection for v1.0)? Or wire through ESO → k8s Secret → `envFrom` like all other secrets?

### 5. Edge case mitigations format
- **Current state**: Success criterion 5 requires documented mitigations for cold-start races, secret rotation, cluster upgrade paths. Nothing exists.
- **Decision needed**: Where do these live? Code comments in the relevant component? A dedicated `infra/OPERATIONS.md` runbook? Or inline in REQUIREMENTS.md?
