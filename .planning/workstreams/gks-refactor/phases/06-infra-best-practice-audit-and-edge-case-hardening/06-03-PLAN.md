**Step 1 — Add to _SECRET_DEFINITIONS:**
Add one new entry to the `_SECRET_DEFINITIONS` list:
```python
("temporal-db-password", "temporal", "temporal-db-credentials"),
```
This will cause the generic loop to create a GCP Secret Manager secret named `{ENV}-temporal-db-password`.

**Step 2 — Create a custom ExternalSecret with correct key mapping:**
The generic ExternalSecret loop in secrets.py uppercases the slug for the secretKey (e.g., `TEMPORAL_DB_PASSWORD`). But the Temporal Helm chart expects a key named `password` inside the K8s Secret `temporal-db-credentials`. So we need a custom ExternalSecret that overrides the default key mapping.

After the generic `external_secrets` loop (around line 157), add a separate block that overwrites the generic entry:

```python
# ---------------------------------------------------------------------------
# Temporal DB credentials — custom ExternalSecret (D-06, D-07)
# ---------------------------------------------------------------------------
# The Temporal Helm chart expects existingSecret: "temporal-db-credentials"
# with a key named "password". The generic loop would create key
# "TEMPORAL_DB_PASSWORD" which doesn't match. Override with custom mapping.

external_secrets["temporal-db-password"] = k8s.apiextensions.CustomResource(
    "ext-secret-temporal-db-credentials",
    api_version="external-secrets.io/v1",
    kind="ExternalSecret",
    metadata=k8s.meta.v1.ObjectMetaArgs(
        name="temporal-db-credentials",
        namespace="temporal",
    ),
    spec={
        "refreshInterval": _REFRESH_INTERVAL,
        "secretStoreRef": {"name": "gcp-secret-manager", "kind": "SecretStore"},
        "target": {"name": "temporal-db-credentials", "creationPolicy": "Owner"},
        "data": [
            {
                "secretKey": "password",
                "remoteRef": {"key": f"{ENV}-temporal-db-password"},
            },
        ],
    },
    opts=ResourceOptions(
        provider=k8s_provider,
        depends_on=[secret_stores["temporal"], sm_secrets["temporal-db-password"]],
    ),
)
```

This overwrites the generic ExternalSecret for `temporal-db-password` (which was created in the loop) because both use the same dict key. The Pulumi resource name is different (`ext-secret-temporal-db-credentials` vs `ext-secret-temporal-db-password`) so it creates a new resource and the old one will be garbage-collected.

Important: The `ENV` variable is already imported at module level from `config`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Pulumi stack config -> GCP Secret Manager | Credentials moved from version-controlled encrypted config to managed secret store |
| K8s Secret -> Temporal pods | ESO syncs secrets; pods reference via existingSecret and env var injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-6-02 | Information Disclosure | Temporal DB credentials | mitigate | Migrate from Pulumi stack secrets (encrypted but version-controlled) to GCP Secret Manager (managed, audit-logged, rotation-capable) via ESO |
| T-6-02b | Tampering | temporal-db-credentials K8s Secret | mitigate | ESO ExternalSecret has `creationPolicy: Owner` — ESO is sole writer; manual edits are overwritten on next refresh cycle (1h) |
</threat_model>
