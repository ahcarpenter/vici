**Schema migration Job** (`temporal_schema_job`):
Follow the `migration.py` pattern exactly. Key differences from the Alembic Job:
- `name=f"temporal-schema-migration-{ENV}"`, `namespace="temporal"`
- `service_account_name="temporal-app"` (per D-12)
- Auth Proxy init container (native sidecar): identical to `migration.py` but uses `temporal_db_instance.connection_name` instead of `app_db_instance.connection_name`
- Main container uses `_ADMIN_TOOLS_IMAGE`, with `command=["sh", "-c"]`
- The `args` field is a single shell string that runs 6 `temporal-sql-tool` commands sequentially with `&&`:
  1. `temporal-sql-tool --plugin postgres12 --ep /cloudsql/$(INSTANCE_CONNECTION_NAME) --db temporal create-database || true` (idempotent — may already exist)
  2. `temporal-sql-tool --plugin postgres12 --ep /cloudsql/$(INSTANCE_CONNECTION_NAME) --db temporal setup-schema -v 0.0`
  3. `temporal-sql-tool --plugin postgres12 --ep /cloudsql/$(INSTANCE_CONNECTION_NAME) --db temporal update-schema -d /etc/temporal/schema/postgresql/v12/temporal/versioned`
  4. `temporal-sql-tool --plugin postgres12 --ep /cloudsql/$(INSTANCE_CONNECTION_NAME) --db temporal_visibility create-database || true`
  5. `temporal-sql-tool --plugin postgres12 --ep /cloudsql/$(INSTANCE_CONNECTION_NAME) --db temporal_visibility setup-schema -v 0.0`
  6. `temporal-sql-tool --plugin postgres12 --ep /cloudsql/$(INSTANCE_CONNECTION_NAME) --db temporal_visibility update-schema -d /etc/temporal/schema/postgresql/v12/visibility/versioned`

IMPORTANT: The `$(INSTANCE_CONNECTION_NAME)` must be injected as a Pulumi Output. Use `pulumi.Output.concat(...)` to build the shell command string, embedding `temporal_db_instance.connection_name` into the args. The Auth Proxy creates the socket at `/cloudsql/<connection_name>`, so the `--ep` flag receives the socket directory path.

IMPORTANT (Research Open Question 1): `temporal-sql-tool` with `--ep` normally takes a TCP host:port. For unix socket connectivity, the tool may need `--ep localhost -p 5432` with the Auth Proxy running in TCP mode instead. Given this uncertainty, configure the Auth Proxy sidecar in **TCP mode** (not unix socket) for the schema Job only: use `--port 5432` flag instead of `--unix-socket`. This means the main container connects to `localhost:5432`. The `--ep` args become `--ep localhost -p 5432`. Remove the volume mount from the main container (not needed in TCP mode). Keep the volume mount on the Auth Proxy sidecar only if using unix socket mode, otherwise remove volumes entirely for TCP mode.

Revised Auth Proxy sidecar args for TCP mode:
```python
args=[
    "--structured-logs",
    "--port=5432",
    temporal_db_instance.connection_name,
]
```

Revised temporal-sql-tool commands (TCP mode, no socket path):
```
temporal-sql-tool --plugin postgres12 --ep localhost -p 5432 --db temporal create-database || true &&
temporal-sql-tool --plugin postgres12 --ep localhost -p 5432 --db temporal setup-schema -v 0.0 &&
temporal-sql-tool --plugin postgres12 --ep localhost -p 5432 --db temporal update-schema -d /etc/temporal/schema/postgresql/v12/temporal/versioned &&
temporal-sql-tool --plugin postgres12 --ep localhost -p 5432 --db temporal_visibility create-database || true &&
temporal-sql-tool --plugin postgres12 --ep localhost -p 5432 --db temporal_visibility setup-schema -v 0.0 &&
temporal-sql-tool --plugin postgres12 --ep localhost -p 5432 --db temporal_visibility update-schema -d /etc/temporal/schema/postgresql/v12/visibility/versioned
```

No volumes needed for TCP mode — remove the `volumes` and `volume_mounts` from both sidecar and main container.
