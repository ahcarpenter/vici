# Phase 02.3: Migrate Jaeger to v2 and Optimize Tracing Setup - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade the Jaeger tracing backend from `jaegertracing/all-in-one:latest` (v1) to the Jaeger v2 architecture in docker-compose.yml, and extend OTel instrumentation to cover the full async pipeline (Inngest functions, OpenAI/GPT calls, Pinecone upserts, Twilio outbound SMS). No new business logic is added — observability only.

</domain>

<decisions>
## Implementation Decisions

### Jaeger v2 architecture
- **Separate containers**: jaeger-collector + jaeger-query (not a single all-in-one container)
- Port 4317 (gRPC OTLP ingest) and 16686 (Jaeger UI) stay the same — no app config changes needed
- Jaeger v2 config via YAML file (not env vars) — mount config file into containers

### Storage backend
- **OpenSearch 2.x single node** as the trace store
- Security disabled for local dev (`plugins.security.disabled=true`)
- Jaeger collector + query connect to OpenSearch via the Elasticsearch-compatible API
- All four containers in docker-compose: opensearch, jaeger-collector, jaeger-query, (existing) app

### Sampling strategy
- **Always-on (100%)** — sample every span; no traces dropped
- Set **explicitly in code** via `ParentBasedTraceIdRatioSampler(1.0)` or `ALWAYS_ON` in `_configure_otel()` in `src/main.py`
- Not via env var — must be visible in code, not hidden in config

### Instrumentation additions
All four gaps are instrumented manually using `with tracer.start_as_current_span(...)` context managers:
1. **Inngest function execution** — span wrapping `process_message` handler body in `src/inngest_client.py`
2. **OpenAI/GPT calls** — span in `ExtractionService.classify_and_extract()` in `src/extraction/service.py`
3. **Pinecone upserts** — span in `PipelineOrchestrator` Pinecone write path
4. **Twilio outbound SMS** — span around `asyncio.to_thread(twilio.messages.create)` calls in `PipelineOrchestrator`

### Span attribute conventions
- Follow **OTel semantic conventions** where they exist:
  - GPT calls: `gen_ai.system = "openai"`, `gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`
  - Twilio SMS: `messaging.system = "twilio"`, `messaging.destination`
  - Pinecone: custom `db.system = "pinecone"`, `db.operation`
  - Inngest: custom `inngest.event`, `inngest.function`
- Non-standard attrs use dot-namespaced flat format

### Resource attributes
All three are set on the `TracerProvider` `Resource`:
- `service.name` — already set to `"vici"` (keep)
- `deployment.environment` — `"development"` or `"production"` derived from `settings` (truthy `inngest_dev` → development)
- `service.version` — sourced from `GIT_SHA` env var (set in docker-compose.yml via `GIT_SHA=${GIT_SHA:-dev}`)
- Claude adds any additional standard OTel resource attributes that improve Jaeger filtering

### Claude's Discretion
- Exact Jaeger v2 image tags (collector, query)
- Jaeger v2 YAML config file content and mount path
- OpenSearch image tag and heap size settings for local dev
- Healthcheck config for opensearch, jaeger-collector, jaeger-query
- Whether tracer is acquired via `otel_trace.get_tracer(__name__)` per-module or a shared singleton

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/main.py:_configure_otel()` — existing function that creates `TracerProvider`, `OTLPSpanExporter`, `BatchSpanProcessor`; this is where sampler config and resource attributes are added
- `src/config.py:ObservabilitySettings` — `otel_endpoint`, `otel_service_name` already live here; `deployment_env` and `git_sha` fields to be added
- `src/main.py:FastAPIInstrumentor`, `SQLAlchemyInstrumentor` — already invoked in lifespan; new instrumentors follow same call site

### Established Patterns
- Tracer acquired via `otel_trace.get_tracer(__name__)` — use same pattern per-module for new manual spans
- `with tracer.start_as_current_span("name") as span: span.set_attribute(...)` — OTel context manager style
- `asyncio.to_thread()` wraps sync Twilio client — span must wrap the full `to_thread` call, not go inside the thread

### Integration Points
- `src/inngest_client.py:process_message` — Inngest function handler; span wraps handler body, propagates trace context from webhook event if present
- `src/extraction/service.py:ExtractionService` — GPT span goes in `classify_and_extract()` method
- `src/main.py:lifespan` — OTel setup runs here before app start; resource attribute additions go in `_configure_otel()`
- `docker-compose.yml` — `jaeger` service replaced by `opensearch`, `jaeger-collector`, `jaeger-query`; `app` `depends_on` updated

</code_context>

<specifics>
## Specific Ideas

- The `OTEL_EXPORTER_OTLP_ENDPOINT: http://jaeger:4317` env var in docker-compose.yml currently points to the old jaeger all-in-one service; after migration it should point to `http://jaeger-collector:4317`
- `GIT_SHA` in docker-compose: `GIT_SHA: ${GIT_SHA:-dev}` — allows CI to inject the real SHA; falls back to "dev" locally

</specifics>
