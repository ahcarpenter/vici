**Gathered:** 2026-03-08

<domain>
## Phase Boundary

Add a Prometheus container to docker-compose.yml that scrapes the app's existing `/metrics` endpoint, pair it with a Grafana container provisioned with a Prometheus datasource and a pre-built FastAPI dashboard, and add custom Prometheus metrics in the app for GPT call instrumentation, Inngest queue depth, and Pinecone sync queue depth. No new business logic — observability infrastructure only.

</domain>

<decisions>
## Implementation Decisions

### Grafana alongside Prometheus
- **Include Grafana** — `grafana/grafana` container in docker-compose.yml, accessible at `localhost:3000`
- **Provision a basic dashboard** — pre-built JSON dashboard for FastAPI metrics (request rate, latency p50/p95, error rate) via Grafana provisioning files (datasource + dashboard JSON mounted as volumes) — zero manual setup after `docker compose up`
- Grafana datasource provisioned pointing to `http://prometheus:9090`

### Custom metrics
- **GPT call metrics** — Custom `prometheus_client` Counters and Histograms in `ExtractionService`:
  - Counter: `gpt_calls_total` (labelset: `classification_result`)
  - Histogram: `gpt_call_duration_seconds`
  - Counter: `gpt_input_tokens_total`, `gpt_output_tokens_total`
- **Inngest queue depth** — Custom Prometheus Gauge tracking pending/failed Inngest events (mechanism TBD by researcher — may use Inngest API or internal state)
- **Pinecone sync queue depth** — Custom Gauge querying `COUNT(*) FROM pinecone_sync_queue` — early warning if Pinecone writes are backing up

### Prometheus config
- **Scrape interval: 5s** for all targets (higher resolution during development)
- **Scrape targets:**
  - `app:8000` — `/metrics` endpoint (FastAPI app)
  - `prometheus:9090` — self-monitoring
- Config file mounted into Prometheus container as `prometheus.yml`

### Claude's Discretion
- Exact Grafana dashboard JSON (panel layout, colors, thresholds)
- Whether Grafana persists data via a named Docker volume
- Inngest queue depth implementation details (depends on what's queryable)
- Prometheus data retention settings

</decisions>

## Existing Code Insights

### Reusable Assets
- `prometheus_fastapi_instrumentator` already imported and wired in `src/main.py` — `/metrics` endpoint already live
- `prometheus_client` likely available as a transitive dep of `prometheus_fastapi_instrumentator` — can use directly for custom metrics
- `PineconeSyncQueue` SQLModel in `src/extraction/models.py` — COUNT(*) query target for the queue depth gauge

### Established Patterns
- Docker Compose services follow pattern: image, healthcheck, ports, depends_on — new prometheus/grafana services follow same structure
- Security disabled for local services (opensearch uses `DISABLE_SECURITY_PLUGIN=true`) — Grafana should default to admin/admin or similar simple local creds
- Config files mounted as volumes (jaeger uses `./jaeger/collector-config.yaml`) — prometheus.yml and Grafana provisioning files follow same pattern

### Integration Points
- `src/main.py` `create_app()` — where `Instrumentator().instrument(app).expose(app)` lives; custom metric registration should happen at app startup
- `src/extraction/service.py` `ExtractionService` — GPT call metrics instrumenting `classify_and_extract()`
- `docker-compose.yml` — add `prometheus` and `grafana` services, `app` service already exposes port 8000
